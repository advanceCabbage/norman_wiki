# 如何写好 Skill：从触发词到信息层级的实战指南

Skill 写作的难点，不在于把规则写得越多越好，而在于让模型在正确的时机稳定地调用，并按可验证的步骤把事情做完。

本文基于 grill 作者的 `writing-great-skills` 一文翻译、吸收与整理而成，原文链接：[https://github.com/mattpocock/skills/blob/main/skills/productivity/writing-great-skills/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/productivity/writing-great-skills/SKILL.md)。

## 一、先决定：谁来调用这份 Skill

Skill 的调用方式，决定了 `description` 的写法与上下文成本。

### 1.1 模型调用

保留 `description`，模型可以自主发现并触发 Skill。

代价是：`description` 会在每轮对话中占据上下文。此时，它应当是一份面向模型的触发词清单。

### 1.2 用户调用

设置 `disable-model-invocation: true` 后，模型看不到 `description`；只有用户输入 Skill 名称才能调用。

代价是：增加用户的认知负担，人必须记住它的存在和用途。此时，`description` 应当是面向人的一行摘要。

## 二、把 description 写成准确的触发器

### 2.1 保持精简

`description` 会增加上下文负担，因此应尽可能精简。

把 Skill 的**主导词**（leading word）放在前面。`description` 的首要任务是触发调用：只保留触发条件，并让每个分支只保留一个触发条件。

例如：

```yaml
description: 用例评审：评审 `TestCases` 中的 JSON 用例描述。用于功能用例评审请求或 `[评审类型] test-case-spec` 派活协议
```

这里的“用例评审”是主导词，后面明确给出了两个触发条件分支。

### 2.2 善用主导词

主导词是模型预训练中已经存在、简洁但带有行为含义的概念，例如 *lesson*、*fog of war*、*tracer bullets*。

它让模型用一个词锚定一整片行为规则；当它在全文重复出现时，定义会分布在正文中，逐渐形成稳定的理解。

它会两次提高可预测性：

- 在正文中，它锚定**执行**：每次遇到该词，模型都会联想到相同的做法。
- 在 `description` 中，它锚定**调用**：当该词也出现在用户提示、文档和代码中，模型更容易将它关联到此 Skill。

持续寻找可用的主导词，以合并重复表述。例如：

- “快速、确定、低开销”可压缩为 *tight*，即紧凑的循环。
- “一个你确信成立的循环”可压缩为 *red*：测试对 bug 失败就是 red，否则不是。

收益不只是减少 token，更是获得更锋利、更稳定的行为锚点。优先使用模型已经理解的词；自造词必须额外花篇幅定义，无法借用模型既有知识。

## 三、用信息层级控制上下文负担

Skill 的内容可以由**步骤**（steps）和**参考资料**（reference）任意组合：可以全是步骤、全是参考，也可以两者兼有。

### 3.1 Skill 内的步骤

步骤必须有序，并且每一步都应提供完成标准。这样，大模型才知道当前步骤何时真正完成。

严格的步骤要求与完成标准，能避免模型过早结束任务。

### 3.2 Skill 内的参考资料

参考资料用于承载按需查阅的定义、规则与事实。

将参考资料按文件分类，并在 `SKILL.md` 中建立索引关系，能让顶层保持清晰。模型可以根据判断分支加载对应内容，实现渐进披露，并减少上下文负担。

## 四、什么时候该拆分 Skill

### 4.1 按调用拆分

当某一能力拥有独立的主导词、应被模型单独触发，或必须供其他 Skill 调用时，应拆成模型调用型 Skill。

代价是：新增一个始终加载的 `description`。

### 4.2 按顺序拆分

当后续步骤，也就是完成后的步骤（post-completion steps），会诱使模型草草结束当前步骤时，应把步骤序列切开。

让后续步骤暂时不在眼前，能促使模型为当前任务做更多前置工作。

## 五、六种常见失效模式

### 5.1 过早完成（premature completion）

当前步骤尚未完成，模型就把注意力转向“结束”。先把完成标准写得更明确、可检查；只有当标准天生无法明确，并且确实观察到仓促行为时，才通过拆分隐藏后续步骤。

### 5.2 重复（duplication）

同一含义出现在多个地方，会增加维护与 token 成本，并不当地抬高该信息的重要性。

这与有意重复主导词不同：前者重复的是完整含义，后者重复的是单个行为锚点。

### 5.3 沉积（sediment）

因为“添加安全、删除有风险”而不断积累的过时内容。如果没有定期精简，Skill 默认会变成这样。

### 5.4 蔓延（sprawl）

即使每一行都有效且不重复，文件仍可能过长。它会降低可读性和可维护性，并浪费 token。

优先将参考资料下沉；再按判断分支或步骤序列拆分。

### 5.5 无效指令（no-op）

模型默认就会做的事，被重复强调却没有改变行为。例如，模型本已“还算仔细”，再写“要仔细”几乎无效；应改为更有力度、能改变行为的词，如 *relentless*。

是否属于无效指令取决于模型的实际默认行为，应通过运行验证，而不是争论。

### 5.6 否定式引导（negation）

告诉模型“不要做 X”会把 X 放进注意力中心，反而更容易发生。

尽量正向描述目标行为，例如用“注释保持一行”替代“不要写冗长注释”。只有不能正向表达的硬性安全边界，才保留禁止语，并同时给出应采取的正向行为。

## 六、写好 Skill 的实践清单

- 先定义这份 Skill 要稳定的**过程**，而不是要求稳定的输出。
- 判断谁需要发现它：模型或其他 Skill 需要时，使用模型调用型；仅供人手动使用时，使用用户调用型。
- 为模型调用型 `description` 写最少且互不重复的触发分支，并以前置主导词开头。
- 将每个动作写成带可检查、尽量穷尽完成标准的步骤。
- 所有分支都要用的信息留在正文；仅部分分支需要的参考资料，移到有明确触发条件的链接文件。
- 将定义、规则和例外共置，让每个含义有唯一事实来源。
- 删除不相关、过时、重复和不改变默认行为的句子；把否定式提醒改成正向目标。
- 若仍然过长，先下沉参考资料；若当前步骤总被后续任务催促，再按步骤序列拆分。

写 Skill 的核心，不是堆叠提示词，而是设计一套能被正确发现、按需加载、持续验证的行为结构。
