**问题二**：技术方案第 1 条：双 Agent 循环评审
1. "最多三轮"这个数字怎么定的？为什么不是两轮或五轮，有没有做过实验验证三轮是较优解？三轮还不通过怎么办——放弃，还是转人工？
	A：在测试验证阶段，使用 10 个需求文档作为测试任务进行测试用例生成。数据显示经过审核者两轮评审后，在第三轮审核达标的占比为 60%。经过审核者一轮评审，在第二轮达标的占比为 40%。因此我们将评审轮次设定为 三轮，当三轮评审仍然未通过时，通知发起任务的研发同学接入并排查测试用例

2. 审核者助手判断"通过"的标准是什么？如果审核者本身也是一个可能出错的模型，你怎么证明"审核者说通过"这件事本身是可信的？这不就是用一个可能出错的模型去验证另一个可能出错的模型吗？
	A： 首先在 Agent 本身，审核者使用 Claude Opus 4.8 模型，通过身份信息和skill 表明当前 Agent 的功能作用是进行用例评审，并给出具体的评审规则和维度。
	 其次：Agent 需要严格按照评分机制给测试用例打分，仅当测试用例得分大于 95 分时才允许审核通过。评分标准为以下：
	 - 全面性 70% （参数类型正常&异常、超长字符、合法枚举&非法枚举、异常 Case、连续多次调用、高并发、边界场景（文件不存在等）、跨端兼容场景（不同端、低端机））
	 - 有效性 30%  （用例写入的文件位置正确、用例的可执行步骤清晰、用例的可测试环境正确、满足 TC+EX 语义、JSON 格式正确）

3. 实际运行中平均要几轮才能通过审核？有没有统计过"三轮都通不过"最终转人工处理的比例？
	   A：实际运行中 60%需要三轮审核、40%需要两轮审核。三轮都不通过的概率较少，月维度偶现 1~2 个三轮审核不通过，原因是：需求文档给的内容太长，通常是事件性 API，同一个需求文档中包含 4~5 个事件型 API。举例：流式上传 API，包含：启动流式分片上传、停止分片上传、取消流式分片上传

**问题四**：技术方案第 3 条：Skill 挂载
1. "代码与用例一致性比对"这个 skill 具体怎么实现的——纯靠 LLM 语义比对，还是有更结构化的手段（比如 AST 分析、接口签名比对）？纯语义比对的准确率怎么保证？
	A : 对比步骤依次是，通过 SKill 提供脚本获取拉取最新代码仓，脚本获取本次 PR 的变更 diff 文件（使用 git  diff 获取变更文件，并记录到 code.diff 文件中），传入 diff 变更文件和对应的测试用例执行代码。判断测试用例是否完全覆盖 diff 文件的变更逻辑。最终生成差异报告，给出差异明细要求人工确认
	- 以代码为准- 代码正确，补充或修改测试用例
	- 以用例为准- 用例正确，代码需修改，PR 不合，由人工修改代码实现
2. 两个 Agent 的 skill 列表里都有"需求解析"，这是各自独立实现的，还是共享同一份能力只是分别挂载？
   - 共享同一个 SKill 实现，只是分别挂载
