## 二、 AI 基于简历项目的提问

##### 2.1 设计 API 知识库相关问题
**问题 1**：你说"总结生成 Q"——这个摘要是怎么生成的？是每篇文档过一遍大模型让它自己总结，还是有固定的模板去抽取入参、出参这些结构化字段？如果是模型自由总结，你怎么保证摘要没有丢关键信息、或者没有幻觉出文档里没有的内容？如何验证几百篇 API 总结出来有没有问题？有没有经过人工抽查？抽样比例是如何的？如果抽查到有几条问题、有几条 API 出问题的时候，你们是怎么做的？

**问题 2**：为什么要设计成"Q 存摘要、A 存原文、中间建索引关联"这么一个两段式结构，而不是直接把原文档切成小块（chunk）做向量化，走最常见的 RAG chunking 方案？你们踩过 chunking 方案的坑吗，还是一开始设计的时候就直接跳过了那个方案？你们使用的这个双向存储工具是什么？为什么它在向量化的时候会有 1000 字符的限制？为什么在保存答案的时候，整篇文章是没有字符限制的？这个向量化的内容和原文存储是怎么关联起来的？

**问题 3**：如果用户的问题同时命中了两三个不同 API 的 Q，比如问题本身就有点模糊，你的系统怎么决定返回哪个/返回几个 A？有没有出现过"检索到了但答非所问"的情况，你们怎么发现和修的？1你的二次过滤是真的做了吗？你的二次过滤是用什么模型做的呢？怎么确保这个模型产生的过滤内容是高质量的呢？你在做二次过滤的时候，一定会再调用一次大模型，这会延长整体流程的时间，这块是怎么考虑的呢？

##### 2.2 建设工单智能助手相关问题
**问题 1**：你说"判断有效性"——这个"有效性"具体指什么？是判断这条工单的结论是否正确，还是判断这条工单是否有复用价值（比如太个例化的问题就不值得入库）？标准是什么，谁定的？

**问题 2**：增量工单走"人工确认"，存量工单走"大模型批量判断"——这两条路径处理方式不一样，是因为什么考虑？是不是意味着现在这套系统还没办法完全信任大模型自动判断，增量场景必须留人工兜底？如果是，这算不算是这个方案目前的一个局限，长期有没有打算把增量也自动化掉？

**问题 3**：700+条历史工单里，会不会存在同一类问题在不同时间被处理出不同结论的情况——比如去年的工单说 A 方案，但后来发现 A 方案有问题、正确做法是 B？这种前后矛盾的知识，你们怎么处理，会不会导致智能客服给用户出前后不一致的答案？

##### 2.3 基于 Claude Code 搭建远端源码检索能力 相关问题
**问题 1**：为什么用固定关键词"deepsearch"触发源码检索分支，而不是让模型自己判断"这个问题是否需要看源码"？这个设计是权衡了什么？

**问题 2**：Claude Code 具体怎么部署调用的——每次请求现拉起一个 session，还是有常驻服务？端到端延迟大概多少？

**问题 3**：客服助手能拉取源码，有没有权限/安全上的考虑？会不会有不该被暴露的代码逻辑被带出来的风险？

**问题 4**："按 API 名称与客户端定位代码仓"——如果一个 API 名称有歧义，或者逻辑分布在多个仓库/服务里，怎么处理？

**问题 5**："拉取最新分支源码"具体是拉的哪个分支？如果拉到的是还在开发中、有问题的分支，会不会导致回答不准？

##### 2.4 持续迭代相关问题
**问题 1**："监控咨询与回复准确率"，准确率具体怎么衡量的——人工标注，还是用户点赞/点踩这类隐式反馈？

**问题 2**："动态调整阈值"是人工定期调，还是有自动化机制？调整依据和频率是什么？

**问题 3**：向量检索和关键字检索两个通道各自返回结果后，怎么融合排序？两个通道结果冲突时怎么处理？

**问题 4**："文档更新后自动同步到知识库"，这个同步机制怎么实现的——监听变更的 webhook，还是定时轮询？

##### 2.5 成果部分（数据口径）
**问题 1**："工单量降 60%"具体怎么统计的？口径是什么（原始进线量，还是排除掉被智能客服拦截的部分）？

**问题 2**：这个下降有多少能确定是这个项目带来的？有没有排除同期其他因素（比如文档质量本身也在变好、业务量变化）？

**问题 3**："工单拦截率超 50%"怎么定义"拦截成功"？如果 AI 答错了但用户没转人工，这个算不算拦截成功？
