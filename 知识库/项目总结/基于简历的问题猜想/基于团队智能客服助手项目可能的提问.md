**问题 9**：图片是如何在向量数据库被存储的？切片是父子分段的原理是什么？
Friday 的父子分段切割原理是：
- **父段基于小标题题目作为基准**，子段是小标题中的内容作为切割的内容（最长 1000 字符），当段落内容超过 1000 字符时，此时会存在多个子段但是父段都是相同的，检索时优先匹配子段，召回时将子段与其对应的父段内容全部召回。
- **父段使用全文作为基准**，子段依旧会按照文档标题结构进行分段。全文仅产生一个父段，检索时优先匹配子段，匹配到任意子段时召回全文内容
- 优势：避免复杂文档或代码被切片导致语义丢失，提高整体召回率和准确率
- 核心原理：只需对子段进行向量化，并存储子文档与父文档之间的映射关系。确保检索到子段时能返回所有父段内容

## 二、 AI 基于简历项目的提问

##### 2.1 设计 API 知识库相关问题
**问题 1**：你说"总结生成 Q"——这个摘要是怎么生成的？是每篇文档过一遍大模型让它自己总结，还是有固定的模板去抽取入参、出参这些结构化字段？如果是模型自由总结，你怎么保证摘要没有丢关键信息、或者没有幻觉出文档里没有的内容？如何验证几百篇 API 总结出来有没有问题？有没有经过人工抽查？抽样比例是如何的？如果抽查到有几条问题、有几条 API 出问题的时候，你们是怎么做的？

A : 摘要是以 API 为维度，使用脚本调用大模型进行批量生成的。详细步骤为：参与总结的 API 文档内容仅包括：API 功能描述、注意事项、基础信息、入参、出参、错误码等内容，排除：示例代码、Native 基础库、依赖组件等与功能描述无关信息。汇总内容 Prompt：你是一个 API 文档摘要专家。请根据以下 API 文档内容，生成一条用于检索索引的功能场景描述。要求总字数不超过 150 字符，必须包含 API 的功能描述总结、使用场景、关键说明、支持的容器。参考示例：输入文档（addCart）→ 输出：调用购物车加购组件，向购物车添加指定商品，支持外卖/闪购/买菜业务，仅 Android/iOS MSC 容器

	A：保障总结内容质量上是通过调整 Prompt 使得仅 10 条 不同类型 API 满足总结要求后，再批量生成、抽样检查。入向量数据库后使用历史问题进行检索验证

**问题 2**：为什么要设计成"Q 存摘要、A 存原文、中间建索引关联"这么一个两段式结构，而不是直接把原文档切成小块（chunk）做向量化，走最常见的 RAG chunking 方案？你们踩过 chunking 方案的坑吗，还是一开始设计的时候就直接跳过了那个方案？你们使用的这个双向存储工具是什么？为什么它在向量化的时候会有 1000 字符的限制？为什么在保存答案的时候，整篇文章是没有字符限制的？这个向量化的内容和原文存储是怎么关联起来的？

     A：我们在具有大量入参、出参的 API 以及示例代码问题上遇到通过文档结构切分的方式不能检索到全部的 API 片段，导致大模型回复不准确的场景。Q&A 的存储是将对 API 文档总结后的支持场景、用途等进行向量化，对整篇 API 文档原文进行数据库存储。向量化后的 Q 和 A 之间建立映射关系，到检索到 Q 时将 A 返回，此时大模型拿到的 API 的完整文档，对于 A 来说是数据库存储不存在长度限制

**问题 3**：如果用户的问题同时命中了两三个不同 API 的 Q，比如问题本身就有点模糊，你的系统怎么决定返回哪个/返回几个 A？有没有出现过"检索到了但答非所问"的情况，你们怎么发现和修的？

	A：在 API 场景中的确存在多个 API 具备相似功能的场景，例如：getSystemInfo 和 getDeviceInfo 都支持设备品牌、设备型号、客户端平台等字段。当用户提问获取设备型号时 getSystemInfo 和 getDeviceInfo 都将被检索出来。对于两个 API 都支持的场景，我们在 API 的说明中有介绍 getSystemInfo 获取的字段更多，API 性能会更差，因此大模型会在仅需要获取设备型号的场景下推荐使用 getDeviceInfo。对于类似的场景我们会在 API 文档中声明使用场景，推荐的使用场景。

##### 2.2 建设工单智能助手相关问题
**问题 1**：你说"判断有效性"——这个"有效性"具体指什么？是判断这条工单的结论是否正确，还是判断这条工单是否有复用价值（比如太个例化的问题就不值得入库）？标准是什么，谁定的？

	A：对于存量的工单，我们批量获取工单内容，然后随机选择40条调用大模型对工单内容进行总结并判断是否适合入库。大模型得出判断条件后再由处理工单的值班同学判断当前工单处理过程是否需要入库。通过获取到基准数据后不断调整大模型判断是否入库的Prompt，使得大模型对随机的40条判断入库结论与人工一致。一致之后再另外使用了20条工单进行测评。对于存量工单判断是否入库的标准主要为：1. 用户问题及解决结论清晰可见 2. 具备“实际参考价值”，例如类似问题提供排查思路、决策建议 3. API功能不支持时具备预计支持时间等。
	对于增量工单，由值班同学判断当前工单解决结论是否适合入库，若适合入库则手动@工单助理机器人将当前工单内容总结并入库

**问题 2**：增量工单走"人工确认"，存量工单走"大模型批量判断"——这两条路径处理方式不一样，是因为什么考虑？是不是意味着现在这套系统还没办法完全信任大模型自动判断，增量场景必须留人工兜底？如果是，这算不算是这个方案目前的一个局限，长期有没有打算把增量也自动化掉？

	A：大模型判断通过入库的标准和人工判断的准确率大约在 90%。由处理工单者判断当前工单是否入库会更加准确，例如：有些工单是通过zoom会议解决的，此时工单聊天对话框中不会留存解决方案，若值班同学判断工单可入库则会录入实际解决方案。另外值班同学手动录工单成本不高，仅需@工单助手即可

**问题 3**：700+条历史工单里，会不会存在同一类问题在不同时间被处理出不同结论的情况——比如去年的工单说 A 方案，但后来发现 A 方案有问题、正确做法是 B？这种前后矛盾的知识，你们怎么处理，会不会导致智能客服给用户出前后不一致的答案？

	A：基本不会出现 去年说 A 方案今年改为 B 方案的情况，但会存在去年说 A 功能未支持，实际今年工单已支持的情况。目前未做任何处理，实际可以每间隔 3 月拉取一次知识库中的工单，将存在预计 XXX 支持类的工单，进行人工审核，若已支持则从知识库中去除

##### 2.3 基于 Claude Code 搭建远端源码检索能力 相关问题
**问题 1**：为什么用固定关键词"deepsearch"触发源码检索分支，而不是让模型自己判断"这个问题是否需要看源码"？这个设计是权衡了什么？

	A：这样设计的原因是，deepsearch 整个检索流程大约需要三到五分钟。这对于问答系统来说，耗时过长，对用户体验不是很好。但如果是有用户主动说需要进行 deepsearch，他将能接受这个耗时。

**问题 2**：Claude Code 具体怎么部署调用的——每次请求现拉起一个 session，还是有常驻服务？端到端延迟大概多少？

	A: Claude Code 具体是通过在服务器上面部署了一个 Claude Code，并在服务器上面设置了一些 Git 仓库的权限，然后再封装。将这个 Claude Code 的 CLI 工具的 mixed CLI 命令封装到 HTTPS 请求里面。当用户实际发起请求的时候，每一次发起请求都创建一个 session，这个 session 就是当前用户的唯一标识。然后会基于这个 session 去调起 HTTP 服务，再最终调到 Claude Code 上，利用 Claude Code 去拉取、获取信息，最后再完成。端到端耗时在3分钟左右。

**问题 3**：客服助手能拉取源码，有没有权限/安全上的考虑？会不会有不该被暴露的代码逻辑被带出来的风险，<font color="#ff0000">不同用户创建的多个 session 什么时候销毁，不同用户连续调用两次代码检索功能是走的同一个 session 吗？</font>

	A:具备智能助手使用权限的用户，均可通过智能助手访问源码，内部同事使用不存在代码泄露风险。claude code仅支持读代码工具，修改、删除代码工具被禁止。不同用户即创建新session时均会自动拉取一次指定代码仓最新代码。

**问题 4**："按 API 名称与客户端定位代码仓"——如果一个 API 名称有歧义，或者逻辑分布在多个仓库/服务里，怎么处理？

	A:首先是通过这个 API 找到 API 文档；API 文档里面会记录这个 API对应的组件；我们维护了组件和代码仓的映射关系；例如安卓端具备五个代码仓，他对应的组件有16个。假设用户输入的API名称有误则无法找到对应的API文档，假设用户输入API名称有错别字则找到相似的文档让用户二次确认

**问题 5**："拉取最新分支源码"具体是拉的哪个分支？如果拉到的是还在开发中、有问题的分支，会不会导致回答不准？

	A:统一拉的都是Master分支，在此之前我们已经在各端制定规范，Master分支是唯一的线上主分支，代码一定是完整的、无误的。

##### 2.5 成果部分（数据口径）
**问题 1**："工单量降 60%"具体怎么统计的？口径是什么（原始进线量，还是排除掉被智能客服拦截的部分）？

智能客服研发出来之前，每周平均是 50+ 的工单数量。智能客服稳定运行之后，平均每周的工单数量降到 20+。

**问题 2**：这个下降有多少能确定是这个项目带来的？有没有排除同期其他因素（比如文档质量本身也在变好、业务量变化）？


**问题 3**："工单拦截率超 60%"怎么定义"拦截成功"？如果 AI 答错了但用户没转人工，这个算不算拦截成功？

工单拦截率超过 60% 的定义如下：

1. 没有正常拦截的情况：
   用户既向智能客服进行了咨询，又向人工提取了工单。我们认为这样的数据是没有正常拦截的。

2. 成功拦截的情况：
   用户咨询了智能客服，但没有向人工提交工单。我们认为这种情况是成功拦截。

在这个统计口径里面，假设客服回答错了，并且用户也没有转人工，那么其实我们也会算他拦截成功。但这种情况根据我们的数据排查来看是非常少的，因为用户遇到问题一定会去解决且问题日志中很少出现用户存在具体的问题智能客服未解决但 未转人工的情况

**问题 4**："值班人力从 0.8 人降至 0.4 人"这个数字是怎么折算出来的？
值班人员这个折射的数据，是通过向值班同学询问他们的体感工作量得到的，0智能客服 出现之前，需要0.8 的人力和智能客服出现之后需要 0.4 的人力，是由相同的值班同学给出的。
