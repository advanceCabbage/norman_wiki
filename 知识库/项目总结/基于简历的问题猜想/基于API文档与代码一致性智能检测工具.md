**问题三**：技术方案第 2 条：双模块架构

1. "拼装 Prompt"具体怎么做的？Prompt 里包含哪些信息，怎么保证不同 API 之间 Prompt 结构的一致性？
	A：Prompt 拼接是在项目工程中实现的，拼接的时机时并发执行检索之前。动态拼接部分包括：
	- 你是一个经验丰富的{platform}开发工程师
	- 使用 active_the_project 激活{projectName}项目，为{projectName}项目的{API}生成一份详尽的报告
	- 对输入参数、输出参数、错误码是一致的，要求找出嵌套对象的输入参数等
	- 定义输出格式，要求以 MD 格式输出。并且给出 fewshot 参考
	- 三端特定差异部分提示词：
		- iOS 端：参数类型为 NSNumber 时统一输出为 number 等、参数必填性重点看 NSAssert 等断言的使用、在属性声明中查看默认值等
		- Android 端：参数必填性重点看@MISIParamChecker、@NonNull 等、默认值可查看构造函数默认值设置、参数类型为 Integer 统一输出为 number
		- Harmony OS 端：使用 interface 定义查看参数的类型

2. "比对线上文档生成差异报告"，这个比对是纯靠 LLM 语义比对，还是有结构化的字段级 diff？如果是纯语义比对，同一个输入多次跑，结果的稳定性怎么保证？

	A：对比线上文档并生成差异报告，是通过 LLM 进行对比。首先在输出及文档合并阶段将待对比的文档，分别统一为相同格式包含：入参、出参、错误码、示例代码、注意事项。入参按照：属性、类型、必填、默认值、说明 格式梳理。之后调用 LLM 对比所有字段差异，「说明」和「错误码说明」如果是相似则忽略差异，示例代码也按照完善性对比，其余字段类型严格匹配
	
1. CodeBase Server"支持 30+编程语言"，但你们实际业务场景应该主要是 iOS/Android/Harmony 对应的几种语言，"30+语言"是 Serena 本身的能力，还是你们实际验证过的语言数？这算不算把工具本身的能力包装成了项目的能力？

**问题四**：技术方案第 3 条：效率优化

1. "串行改并行"具体怎么实现的？三端并发检索会不会触发代码仓接口的限流？

	A：实现是通过创建多个实例的方式实现并发，原本 Serena 仅仅是单实例，多个任务同时进来时需要排队处理。修改为多实例后，android、iOS、harmony 各一个实例互相不影响，由调用参数决定具体执行的实例，如此可实现多实例并发。**假设两个用户同时发起文档检测时会出现什么情况？，能支持两个用户同时调用吗？**
	- 并行实现，原本 Serena 仅有一个 agent 实例，agent 实例负责激活目标项目、执行工具调用等流程。android、iOS、harmony 属于不同项目，不能使用同一个 agent 进行代码分析。通过支持创建多个 Agent 实例的方式，实现 android、iOS、harmony 均各自独享一个 agent，agent 之间相互独立互不影响，使得三端的代码检测可以并发执行。
	- Serena agent 的记忆机制
		- 以 session 为维度的记忆机制，相同 session 会共享记忆文件，记忆内容包括：会话记录、每次 agent 的运行记录、会话摘要、用户长期记忆
		- 以项目为维度的记忆机制，相同的项目可以共享，存储在.serena/memory 目录下，和项目相关的记忆文件
	- 多人同时触发代码检索任务时，通过锁机制，排队执行。否则同一个 agent 可能出现串激活项目、工具执行等问题，一个 API 从分析到 diff 对比完成大约耗时 5 分钟。

1. "预初始化记忆文件"具体是什么内容？这个缓存要花多久建好，代码仓更新之后缓存是不是就过期了，过期了怎么处理？
	A：「预初始化的记忆文件」包含的内容为：
	- project.yml 文件：包括：projectName：msi-android、 language：java、read_only:true、ignored_paths:[]、excluded_tools:[]、initial_prompt
	- memories 文件夹：
		- code_style_and_conventions.md:代码风格约定、注解使用（@msiApiMethod）
		- project_overview.md : 项目描述、项目文件结构、技术栈
		- suggested_commands.md: 项目涉及的命令，构建命令、测试用例、Git 相关命令
- 从以上文件来看，记忆文件中主要是包含这个项目的通用型描述，它不太会随着开发增加一些 API 或者实际功能的开发而变更。因此，它的变更频率是极低的。
