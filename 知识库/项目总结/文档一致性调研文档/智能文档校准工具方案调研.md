## 一、背景

当前MSI官网文档存在部分文档与源码实现不一致问题。面对存量API文档和增量API文档需求改动，由人工校准文档与三端源码一致性工作量巨大。**因此期望借助AI实现分析三端源码、与官方文档校验、生成Diff差异文档全自动校验文档差异**

**现状描述**：现有校准存量API文档差异问题的方式是借助AI编辑器（Cursor、augment、Continue等）的CodeBase能力、结合MCP引导大模型执行代码分析流程，属于是半自动方案详细方案见：[https://km.sankuai.com/collabpage/2711655882](https://km.sankuai.com/collabpage/2711655882)。现有方案无法全自动校准所有文档与源码差异，无法做到增量更新需求时自动校准源码与文档差异，**原因是现有方案强依赖AI编辑器的CodeBase能力完成对代码仓库检索**

**需求描述**：期望通过自建CodeBase能力实现对代码仓库检索能力，结合Agent满足对增量和存量需求自动批量校准基础API与文档差异

## 五、附录：术语说明

#### 5.1 Merkle 树

Merkle 树是一种树状结构，其中每个"叶子"节点都标有数据块的加密哈希值，每个非叶子节点都标有其子节点标签的加密哈希值。这创建了一个层次结构，通过比较哈希值可以高效地检测任何层级的更改。

#### 5.2 Tree-sitter

[Tree-sitter](https://tree-sitter.github.io/tree-sitter/)是一种解析器生成工具和增量解析库。它可以为一个源文件构建一个具体语法树，并在源文件编辑时高效地更新语法树。Tree-sitter 的目标是：

- 足够通用，可以解析任何编程语言
- 足够快速，能够在文本编辑器的每个按键时进行解析
- 足够健壮，即使在存在语法错误的情况下也能提供有用结果
- 无依赖性，因此运行时库（用纯 C 编写）可以嵌入到任何应用程序中

#### 5.3 .scm查询模式

.scm查询模式是对语言定义风格的总结，对于不同语言存在不同的.scm查询模式定义，以TypeScript举例：

```scheme
; 函数定义查询
(
  (comment)? @comment
  (function_declaration
    name: (_) @name
    parameters: (_) @parameters
  ) @definition
)

; 方法定义查询
(
  (comment)? @comment
  (method_definition
    name: (_) @name
    parameters: (_) @parameters
  ) @definition
)
```

#### 5.4 trigram

trigram是一种文本分词技术，特别适用于全文搜索引擎，核心原理是将一行代码从头到尾每三个字符组成一个词语。举例：

```JavaScript
// Trigram分词示例
const text = "getUserById";

// 传统的trigram分词过程：
const trigrams = [
  "get",  // 位置0-2
  "etU",  // 位置1-3
  "tUs",  // 位置2-4
  "Use",  // 位置3-5
  "ser",  // 位置4-6
  "erB",  // 位置5-7
  "rBy",  // 位置6-8
  "ByI",  // 位置7-9
  "yId"   // 位置8-10
];
```

#### 5.5 LSP（语言服务器协议）

LSP（语言服务器协议）：Language Server Protocol (LSP) 是微软在2016年开发的一套开放标准协议，用于在代码编辑器/IDE与语言服务器之间建立通信。它解决了传统开发工具中"M×N问题"，即M个编辑器需要支持N种语言时，需要开发M×N个插件的复杂性。

#### 5.7 Token消耗量大

Token消耗量大体现在以下几个方面：

1. 复杂的系统提示词
2. 文件加载策略不够智能（例如：node_modules）
3. 上下文窗口管理机制（自动注入大量项目上下文）
4. 多轮对话累积效应（传入历史对话内容）
