## 项目背景

我所在的团队MSI（美团标准接口）负责为全公司提供可调用的API基础能力，能力范围覆盖三端、美团系APP及美团内部多容器场景业务。公司级API能力用户量巨大，常规客服群有1000+人，未包含部分提工单的用户未在客服群。大量的用户群体导致工单量工作大，当日值班同学需消耗1PD人力在处理工单上。因此需要建设智能客服机器人解决人力紧缺问题
## 核心收益

- 工单数量平均每周约50个，降低到每周约22个，减少共工单数量每周约60%，减少客服工作量约为50%
- 智能客服每周使用人次约44，其中再次提工单人数约为12，工单拦截率为73%
- 深入研究RAG检索机制，并于终端技术中心分享对RAG的应用实践
## 核心动作
1. 将MSI所有知识内容分类
	1. API文档类型通过双通道索引架构向量化，将每篇API文档总结，将总结内容作为Q，将整篇文档作为普通存储，也可称为Q&A架构，对总结内容向量化，对Question进行向量化，Answer用作普通存储，Q和A之间建立索引关系。向量检索匹配到Q时，同时将A一起返回
	2. 接入指南、开发指南采用基于文档结构进行分类
	3. 将存量工单、增量工单利用大模型进行总结，将有效的工单向量化并入库
	4. 接入工作流模式，为机器人增加代码检索能力
		1. 识别用户提供的API、端（Android、iOS、Harmony）。依据API推断代码仓git地址
		2. 远端部署Claude code，拉取对应代码仓并基于用户问题进行代码检索分析
		3. 同步触发知识库检索
		4. 汇总代码分析和知识库检索答案，再由大模型回复内容
		5. 代码检索触发时机：用户提交工单智能机器人主动回复第一轮回复；用户输入deepresearch时触发代码检索任务
2. 持续统计并优化智能机器人，依据后台数据即用户咨询智能机器人后又去提TT场景
	1. API介绍文档不够完善、有误
	2. 引导用户完善问题，通常用户仅仅说问题现状，不会指明具体API名称。例如：用户提问：线上无法获取到数据，但测试环境能正常获取数据。此时需要引导用户提供具体的API名称和对应的客户端
	3. 智能客服回答的问题范围太广。例如：用户提问：uploadFile上传视频显示成功，但上传成功的URL无法访问。此时机器人回复：可能原因是 1. 可能状态码非200但仍然进入成功回调 2. 视频文件数据不完整 3. S3 桶权限配置问题。真实的原因是S3桶权限配置问题也是最可能的原因，此时机器人给出多个原因会误导用户，也会降低用户看原因的耐心，导致该工单重新被人工介入
	4. 智能客服机器人无法处理的问题:例如第三方依赖SDK报错导致的问题
3. 初期模型能力对比实验测试，对比calude3.5、gpt-4o-mini、qwen-vl-max-latest、Kimi等模型效果
4. 智能机器人提示词调优
5. 建设API文档更新自动同步到知识库机制、推动文档完善
