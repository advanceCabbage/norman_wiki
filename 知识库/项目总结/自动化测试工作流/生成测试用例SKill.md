# EC 用例描述设计规则

本文档指导 `generate-ec-test-case-spec` skill 如何根据 API 文档生成 EC 平台的 JSON 用例描述。产物是 **JSON 描述**(场景层),不是可执行 JS 代码

JSON 文件的字段结构与样本格式见同目录 `ec-json-schema.md`。本文档专注于"应该设计哪些 case、覆盖哪些场景"。

测试用例分为 **canIUse**、**通用**、**场景** 三大类。针对不同类型的 API 可参考项目中 getImageInfo(异步桥)、getStorageSync(同步桥)、onLocationChange(事件桥)、FileSystemManager.access(通过上下文调用的桥)的完整 JSON 描述。

---

## 1. canIUse 测试用例

canIUse 是一个额外的测试用例,根据 API 文档中 API 的入参和出参信息生成,用于判断 API 和 API 的入参、出参是否支持当前环境。如果 API 只支持部分平台,canIUse 用例只需要校验 API 所支持的平台。

### 1.1 硬约束

项目中 `msc/AI_TestCases/canIUseAll/index.js` 由 `generate-canIUseAll.js` 脚本自动生成,会从每个 API 的用例文件中提取**第一条**用例作为 canIUse 汇总校验入口。该约束同样要求 EC JSON 描述阶段就预留位置,确保后续代码生成时能落到首位。因此每个 API 的 JSON 描述**必须**满足:

1. **schema 一律写进 TC 文案,EX 下保持空对象 {}**:不再把 `schemaList` / `negativeSchemas` / `platformBranches` 等结构化数组罗列在 EX 下;canIUse 因此**不再是** §3.2 占位结构的例外
    
2. **TC 命名统一为 TC: [在 <平台> 平台]调用 canIUse 校验，传入参数为schema1,schema2,…** ——把该条用例要校验的 schema **逐个用英文双引号包裹、半角逗号分隔**列进文案;全平台一致时省略 `在 <平台> 平台` 前缀
3. **EX 命名统一为**:可用 `EX: canIUse 返回成功`,不可用 `EX: canIUse 返回失败`(两者 EX 下均为 `{}`)
    
4. **全平台正向用例(必有)放 标准用例 首位**:只列出在 API 所有受支持平台都可用的 schema → `EX: canIUse 返回成功`。canIUseAll 提取的首条即这条
    
5. **负向用例(仅当存在应返回 false 的 schema 时)**:列出这些 schema → `EX: canIUse 返回失败`;没有则省略
    
6. **平台差异单独列举**(详见 §1.3):某 schema 只在部分平台可用时**不要**塞进第 4 条的全平台正向用例,而是**每组单独成条**——TC 文案写明支持的平台(如 `在 Android、iOS 平台`)并列出这些 schema,`EX: canIUse 返回成功`(在不支持的平台上该条按平台 skip)
    
7. **JS case key**:首条正向 = `<API名称>_canIUse`,其余 canIUse 用例追加区分后缀(如 `<API名称>_canIUse_negative` / `<API名称>_canIUse_notIOS`,API 名中的点号在 case key 中替换为下划线)
    
8. **canIUse 汇总列表必须递归到 ## 复杂对象 章节定义的所有嵌套子对象的子孙字段**,直到子孙字段都是基础数据类型为止
    

> 漏掉任一嵌套子对象的子孙字段都视为 canIUse 描述不完整。生成 canIUse 描述前必须先把 `## 复杂对象` 章节里所有嵌套类型读完并展开为 canIUse 查询链。

### 1.2 入参出参查询链规则

JSON 的 canIUse 描述里需要列出所有要校验的 schema 字符串。schema 命名规则:

|角色|API 类型|schema 形式|示例|
|---|---|---|---|
|入参|全部|`<api>.object.<字段>`|`getImageInfo.object.src`|
|出参|异步 API|`<api>.success.<字段>`|`getImageInfo.success.width`|
|出参|同步 API|`<api>.return.<字段>`|`getSystemInfoSync.return.screenWidth`|
|出参|事件 API|`<api>.callback.<字段>`|`onLocationChange.callback.latitude`|

**特殊情况(canIUse 不校验)**:

- **入参里的回调函数字段**(`success` / `fail` / `complete`):异步 API 中一定存在,且不支持 canIUse 校验
    
- **出参中的 errMsg / errno**:通用字段,不支持 canIUse 校验
    

### 1.3 平台判断(6 条)

递归遍历所有入参和出参的属性及其子孙属性,分析每个参数所支持的平台(iOS、Android、HarmonyOS):

1. 参数属性默认支持每个环境
    
2. 属性说明中明确"只支持 X"或"不支持 X" → 按属性说明
    
3. 最低支持版本中标明不支持某平台 → 该属性不支持该平台
    
4. **"仅 X 平台支持"或"X 平台不支持"的字段**:按 §1.1 第 6 条**单独成条**,TC 文案写明支持平台并列出该字段。示例:onLocationChange 的 `traceId`、`wifiInfo`、`indoorScore` 仅 Android 支持 → `TC: 在 Android 平台调用 canIUse 校验，传入参数为"onLocationChange.callback._mt.traceId","onLocationChange.callback._mt.wifiInfo","onLocationChange.callback._mt.indoorScore"` / `EX: canIUse 返回成功`
    
5. **"灰度中"状态的字段**:最低支持版本标注 `<版本号>(灰度中)`(如 `iOS 组件: 12.41.200(灰度中)`)→ 在该平台上视为不可用,同样按 §1.1 第 6 条单独成条。示例:openCity 的 `dpName`、`dpPinyin`、`dpDistrictId` iOS 灰度 → `TC: 在 Android、HarmonyOS 平台调用 canIUse 校验，传入参数为"…dpName","…dpPinyin","…dpDistrictId"` / `EX: canIUse 返回成功`
    
6. **枚举值在单平台不支持**:某枚举字段的**部分枚举值**在特定平台不支持(如 encrypt 的 `encryptType=4`(TTE 国密)在 HarmonyOS 不支持,但 0-3 全平台支持):
    
    - 该枚举值对应的 case 描述应显式说明"仅 Android 和 iOS 平台执行,HarmonyOS skip"
        
    - canIUse 描述无需对"该枚举值"做条件分支(canIUse 校验的是字段维度,不是字段值维度)
        
    - 错误场景仅在不支持的平台才会触发"枚举值不支持"的错误码,需要根据平台判断预期错误码

## 2. 通用测试用例(按入参类型)

通用测试用例根据 API 文档中入参信息生成,需要**递归遍历每一个入参及其子参数**,根据参数类型与是否非必填来设计测试场景。

### 2.1 类型 × 场景矩阵

每个入参字段按下表对应类型的列设计 case。单元格含义:`必/非` = 必填和非必填都要设计;`必` = 仅必填字段设计,非必填**不要**设计;`—` = 不适用;特殊单元格按表注说明。

|场景|string|number|boolean|enum / 多类型|object|array|callback|
|---|---|---|---|---|---|---|---|
|正确值(给出【具体值】)|必/非|必/非|必/非 ①|每个枚举值 必/非 ②|必/非|必/非|必/非|
|正确类型但不符合字段说明|必/非|必/非|— ③|— ③|—|—|—|
|`null`|必|必|必|必|必|必|—|
|`undefined`|必|必|必|必|必|必|必 ④|
|类型错误(例:string→123)|必|必|必|非枚举值|必(含数组撞车 ⑤)|必|—|
|不传该属性|必/非|必/非|必/非|必/非|必/非|必/非|—|
|空值|空字符串 必|`0` 必|—|—|空对象 `{}` 必/非 ⑥|空数组 必/非|—|
|超长 / 超大|≥10000 字符 必/非 ⑦|—|—|—|—|—|—|

**表注**:

- ① **boolean 的"正确值"必须显式拆成 true 和 false 两条独立 case**——只测其中一个就漏掉相反取值的分支(常见漏测:`needMediaLocation` 约定为 true 才返回经纬度,只设计 `true` 一条就发现不了 `false` 情况下仍返回经纬度的回归)
    
- ② **enum 的"正确值"按每个枚举值各设计一条**,数组元素是枚举值的补充见 §2.2.3
    
- ③ boolean / enum **不存在"正确类型但不符合"的中间态**(boolean 只有 true/false 合法值;enum 的"不符合"等价于"非枚举值",已在"类型错误"行覆盖)
    
- ④ **事件 API 的入参回调函数(如 listener、callback)默认按"必填"处理**,无论文档是否显式标注必填列。原因:事件 API 文档 `## 入参` 段通常只有 "`function listener` / 需要监听的回调函数" 这样的纯文字描述,没有标准的"必填"列;但事件 API 没有回调就无法监听事件,业务上一定必填
    
- ⑤ **数组撞车场景**:数组在 JS 中 `typeof` 同样返回 `"object"`,但语义上不是 plain object,object 类型字段必须额外设计"传入数组"case(如 `[]` 或 `[1, 2, 3]`),校验 API 是否正确区分 plain object 与 Array
    
- ⑥ **空对象**用于校验 API 对"形状正确但无任何子字段"的兜底——子字段全部非必填时应正常处理;若文档对必填子字段有要求,则应触发对应错误码
    
- ⑦ **超长字符串**(≥10000 字符,可由 `'a'.repeat(10000)` 构造)。与 §3 大数据场景的区别:大数据是 API 维度命中信号才设计(数据量按 1 MB / 5 MB / 10 MB 递增),本条是**字段级**强制兜底,任何 string 字段都要测一次
    

### 2.2 类型特殊补充

#### 2.2.1 number 范围边界

如果文档说明中标注了取值范围(如"取值范围 1-20"、"范围 -90~90"、"最大 100"、"必须为正整数"等),必填和非必填字段都必须设计下列 4 条范围边界 case:

- **下限值**(下限为 1 则传 1;下限为 -90 则传 -90),预期 success
    
- **上限值**(上限为 20 则传 20;上限为 90 则传 90),预期 success
    
- **下限 -1**(下限为 1 则传 0;下限为 -90 则传 -91),预期 fail
    
- **上限 +1**(上限为 20 则传 21;上限为 90 则传 91),预期 fail
    

示例:chooseImage 的 count 字段标注"取值范围 1-20",应设计 `TC: count 传入下限值 1` / `TC: count 传入上限值 20` / `TC: count 传入 0(下限-1)` / `TC: count 传入 21(上限+1)`。

#### 2.2.2 number 超时字段

字段名为 `timeout` / `*WaitTime` / `*Timeout` 等超时类语义(如 request 的 `timeout`、getLocation 的 `gpsWaitTime`):除矩阵场景外,传一个明显过小的值(如 1 ms 或文档默认值的 1/100)应触发文档中的超时错误码,对应失败 case 见 §3.3 错误码覆盖原则。

#### 2.2.3 array 元素枚举值补充

如果数组元素是枚举类型(如 chooseImage 的 `sizeType: Array<string>` 元素取 `original/compressed`,scanCode 的 `scanType: Array<string>` 元素取 `barCode/qrCode/datamatrix/pdf 417`),除矩阵场景外,必填和非必填字段都必须额外设计:

- 数组包含**单个合法枚举值**(如 `['original']`),预期 success
    
- 数组包含**全部合法枚举值**(如 `['original', 'compressed']`),预期 success
    
- 数组包含**单个非法枚举值**(如 `['invalid']`),预期 fail
    
- 数组包含**混合合法 + 非法枚举值**(如 `['original', 'invalid']`),预期 fail
    
- 数组包含**重复合法枚举值**(如 `['original', 'original']`),按文档说明判断预期;文档未说明则预期 success
    

### 2.3 实施建议

#### 2.3.1 入参基线 + 增量原则(通用,所有 API 强制)

矩阵用例**一次只变一个变量(OFAT)**:被测字段取该 case 的值,其余字段全部回落到一组固定的合法"基线值"。为了让"其余字段取什么"在描述层可读、可评审、可被代码生成稳定翻译,**必须把基线显性化**,而不是留给读者脑补:

1. **基线唯一、按 API 实例化**:每个 API 的 `入参` 下定义**一条** `_入参基线`(结构见 `ec-json-schema.md` §4.1),给出_所有_字段的一组合法默认值——必填→合法值;非必填→默认不传(或注明取的合法值);嵌套对象→连叶子子字段一起给。机制通用,内容因 API 而异。
    
2. **TC 只写增量**:任意字段、任意 case,只描述「被测字段 + 取值」,并以 `(其余取基线)` 收尾;**不要**在每条 TC 里重复罗列其他字段(近百条 case 全抄一遍既冗余又是改一处动百处的维护负担)。
    
3. **负向用例保持其余合法**:失败类 case,未被测字段必须是基线合法值,保证失败可**归因**到被测字段——否则 `errno` 断言站不住(失败可能由别的字段引起)。
    
4. **有依赖的字段强制内联**:判定标准——**只要"改这个字段会牵动另一字段的合法性 / 是否必填"**(互斥/二选一、条件必填、字段联动),光靠"取基线"表达不了,TC 必须写清相关字段状态,并在 §3.1「入参字段联动」下补组合 case。
    
5. **退化与省略**:只有 1 个可组装入参(如事件 API 的 `listener`)或零入参的 API,没有"其余字段"可固定 → **省略** `_入参基线`;只有 ≥2 个入参字段时才必须写。
    

#### 2.3.2 其他

- **禁止设计回调字段校验 case**:切记不要设计 `success` / `fail` / `complete` 回调函数校验的 case,这些回调函数在异步 API 中一定存在,不需要再设计

## 3. 测试用例类型 T 01-T 11

EC 评审从 11 类用例类型(T 01-T 11)看 API 的覆盖广度;每类对应 §2 通用矩阵或 §3.1 / §3.5 场景规则。LLM 根据 API 文档判断"应命中"哪几类,未命中类**不强行凑数**(EC 8 评审会接受合理不命中)。

|编号|类型|设计规则位置|触发信号(命中即应该有)|
|---|---|---|---|
|T 01|正向|§2 矩阵"正确值" + "枚举值正确值"行|所有 API(强制)|
|T 02|反向|§2 矩阵"类型错误 / null / undefined / 不传 / 空值 / 超长"等行|所有 API(强制)|
|T 03|异常|§3.1 异常场景 + §3.5 异常场景|文档列错误码 / 工单含异常 / 状态机调用|
|T 04|边界|§3.1 边界场景 + §2.2.1 number 范围 + §2.2.3 数组枚举|入参含范围 / 集合 / 枚举边界|
|T 05|流程|§3.1 入参联动 + §3.1 API 联动|多步业务流程 / API 间状态依赖|
|T 06|权限|§3.5 权限场景|文档声明权限 / 操作敏感资源 / 错误码 59995-60001|
|T 07|安全|§3.5 安全场景|加密 / 注入风险 / 敏感数据|
|T 08|性能|§3.5 性能场景 + §3.5 大数据场景|响应/内存约束 / 入参出参大体积 / 高频调用|
|T 09|兼容性|§3.5 跨环境兼容场景|文档明示网络 / 系统版本 / 机型差异|
|T 10|幂等 / 重试|§3.1 多次连续调用 + §3.5 并发场景|写操作 / 状态变更 / 共享资源|
|T 11|降级与容错|§3.5 降级容错场景|依赖外部 SDK / 网络 / 本地资源|

**编号与设计规则的关系**:T 01-T 11 是**评审视角**的分类(用于 EC 8 + 评审报告 §2 节);**设计视角**仍按 §2 矩阵 + §3.1 / §3.5 章节组织。LLM 设计时直接按规则章节走,LLM 评审时按 T 编号统计覆盖率。

### 3.1 五类常见场景(对应 T 03 / T 04 / T 05 / T 10)

- **多次连续调用**:同一 API 连续调用 5 次(或多次),验证幂等性、副作用、状态一致性
    
- **入参字段联动**:多个入参字段之间存在联动关系(互斥/二选一、条件必填、取值相互影响)时的组合验证。承接 §2.3.1 第 4 条——这类字段在单字段矩阵里靠"取基线"表达不了,必须在此单独设计组合 case(如认证字段 A、B 二选一:只传 A / 只传 B / 都传 / 都不传,各自的预期)
    
- **API 联动**:多个 API 配合使用,如 setStorage→getStorage、startLocationUpdate→onLocationChange、open→read→close
    
- **边界场景**:文件不存在 / 已存在、key 不存在 / 已删除、达到取值上下限等
    
- **异常场景**:无权限访问、超时、参数非法等
    

### 3.2 设计原则(4 条)

1. API 用例执行环境中已默认开启所有权限和服务,APP 执行状态正常,**不要**设计超出当前环境能力的场景用例,**不要**自己 mock 数据
    
2. 部分错误码场景无法通过当前环境能力模拟,无需设计对应场景用例(详见 §3.3 第 2 条)
    
3. **允许并鼓励**设计跨环境兼容性测试,但需限定到**网络环境 / 系统版本 / 机型**三个维度(详见 §3.5 跨环境兼容场景)。case 描述里必须显式标注目标环境;**不要**笼统写"测试兼容性"或"跨端测试"。**不同应用宿主、不同客户端**的差异不在 case 设计范围(三端差异由 canIUse 的平台差异用例表达,不需要 case 化)
    
4. **不要**设计需要手动执行的场景用例(如手动模拟用户登录、手动点击)
    

### 3.3 错误码覆盖原则

文档 `## 错误码` 章节列出的每条错误码都应尝试设计对应的失败 case 描述,不要只覆盖通用错误码(如 29999)而漏掉业务场景错误码:

1. **能在当前环境模拟的错误码必须设计 case 描述**:
    
    - 参数非法类(如 29999 filePath 为空 / flag 非法)→ 通用用例覆盖
        
    - 业务场景类(如 20007 r 模式打开不存在文件、20004 ax/wx 模式打开已存在文件、20003 a/a+/w/w+ 模式打开是目录、59995 无权限访问等)→ 场景用例覆盖
        
    - 超时类(如 request 的 10002 "超时"、uploadFile 的 10002 "上传文件超时")→ 通过 §2.2.2 传小值 timeout 触发,或配合大数据场景自然触发
        
2. **无法模拟的错误码按 §3.2 第 2 条跳过**,但必须在 JSON 顶层 `_未覆盖错误码说明` 字段里显式说明(格式见 `ec-json-schema.md` §7;典型条目形如 `"20006 (iOS) 文件句柄创建失败:..."`)
    
3. **错误码与平台对应关系**严格按文档列出的平台。某错误码只在部分平台返回时(如 59995 仅 Android/HarmonyOS 返回),case 描述里必须用文字限定校验平台(如"仅 Android 和 HarmonyOS 上断言 59995")
    

### 3.4 使用注意事项业务约束转化

API 文档 `## 使用注意事项` 章节描述的业务约束必须转化为对应的场景测试用例描述。常见类型与转化方式:

|约束类型|示例|转化方式|
|---|---|---|
|版本限制|"PUT 方法安卓 / iOS 12.53.200 开始支持"|一般无需 case(框架默认在最低版本以上运行);若约束影响入参合法性则补充验证 case|
|平台特定行为|"iOS 12.48.200 起 encryptType=1 时 iv 必传,否则返回 20002"|必须设计仅 iOS 生效的 fail case,断言不传 iv 返回 20002|
|字段联动|"needMediaLocation 为 true 时返回信息才包含 latitude / longitude"|必须设计字段联动 case,验证联动行为|
|大小 / 长度限制|"iOS TTE 明文大小超过 2 MB 返回 20015"|必须设计超限 fail case(若环境能模拟,见 §3.5 大数据场景)|
|跨端行为差异|"HarmonyOS 对 AES key 做截取和补齐处理,iOS/Android 使用原始 key"|**不**设计 case(§3.2 第 3 条:不同客户端差异不是 case 范围)|

**推导原则**:注意事项里每一条业务约束都应判断"是否可转化为可执行的 case",能转化的必须设计;只描述跨端差异、版本演进等不可独立测试的内容不强制转化。

### 3.5 高级场景类型(对应 T 03 / T 06 / T 07 / T 08 / T 09 / T 10 / T 11,命中信号才设计)

下面 8 类场景**不是所有 API 都设计**,只有命中触发信号才设计对应 case。命中后必须在 `场景测试` 下用对应分类名挂 case。

#### 3.5.1 设计要点(所有高级场景共用)

- case 描述里**必须显式标注**目标环境 / 平台限定 / 数据量 / 次数(如"仅 Android 低端机执行"、"WiFi 切移动网络中途"、"iOS 17 以下"、"连续 50 次"、"传入 1 MB 字符串")
    
- 后续代码生成阶段会读这些标注翻译为 `skip: util.isXxx` / `Promise.all` / `Array.from` 等代码结构
    
- 产生资源占用的 case(如 SocketTask 连接、setStorage 后、大文件读取)必须在描述里包含"清理副作用"说明,后续代码生成会翻译为 `after` 钩子
    
- 三端默认都能跑,**不**加 `_暂不可执行: true` 标注;笼统的"跨平台测试" / "兼容性测试" 不允许,必须落到具体维度
    
- 各场景分类放在 `场景测试` 下,**并发场景放在末尾**(便于代码生成阶段读次数);任一类本轮未设计则**整类省略**
