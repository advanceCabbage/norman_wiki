# EC 用例评审：规则与报告模板

本文件是 [`评审EC用例SKill.md`](评审EC用例SKill.md) 的按需参考资料，是 EC 评审规则、评分和报告格式的唯一事实来源。评审开始执行 EC 1–EC 13、计算评分或输出报告时必须读取本文件。

## 证据与范围规则

### 评审输入

评审对象为 `msc/EC_TestCases/<module>/<apiName>.json`。有效范围依次为：

1. `[变更文件]` 中位于 `msc/EC_TestCases/<module>/` 的 `.json`；
2. `[模块]` + `[API名]` 推导的 `msc/EC_TestCases/<module>/<apiName>.json`；
3. 分支 diff 中的同类文件。

分支链接只接受可解析出 `branch=refs/heads/<branch>` 的链接。`maxRounds` 固定为 `3`，只透传，不改变。评审依据为需求/API 文档、目标分支 JSON 和设计规则速查表；diff 只用于定位文件，不能用来推断覆盖率。

### 证据格式

每个失败项使用以下格式：

```text
<相对路径> : <JSON 路径> | <不超过 120 字的 JSON 节选>
```

证据不能支持判断时，标记 `n_a` 并说明缺失的输入；不把猜测写成失败。

## EC 1–EC 13 评审规则

### 全面性

| ID | 通过条件 | 失败信号 |
| --- | --- | --- |
| EC 1 canIUse 覆盖 | `标准用例` 首项只有一条合规 canIUse TC；`schemaList` 递归覆盖全部入参与出参字段；文档存在三端差异时含 `platformBranches`。 | 缺失、多条、位置或命名不合规、漏嵌套字段、漏平台分支。 |
| EC 2 通用用例 | 每个入参按类型矩阵有必需 case；每个出参（含嵌套字段）有 TC；EX 区分“必然存在”与“若存在”。 | 入参、类型补充、出参或嵌套字段漏测。 |
| EC 3 场景用例 | 至少一类常见场景；命中并发/大数据信号时有对应分组；不可模拟的文档错误码有顶层 `_未覆盖错误码说明`。 | 场景为空、命中信号却缺分组、遗漏应说明的错误码。 |
| EC 4 用例类型 | 对 T01–T11 中按 API 文档应命中的类型逐项判定；实际命中比例不少于 80%，覆盖识别 iOS、Android、HarmonyOS 差异。 | 应命中类型缺失，或比例低于 80%。 |

### 正确性

| ID | 通过条件 | 失败信号 |
| --- | --- | --- |
| EC 5 文件位置 | 路径严格为 `msc/EC_TestCases/<module>/<apiName>.json`；模块和文件名与 API 文档一致，点号保留。 | 目录、模块或文件名不符。 |
| EC 6 JSON schema | JSON 可解析；顶层仅一个 API 名 key；含 `标准用例` 与 `场景测试`；`标准用例` 含 `canIUse`、`入参`、`出参`；TC/EX 前缀合规。 | 解析失败、顶层或分组缺失、TC/EX 格式错误。 |
| EC 7 模块与文件 | 一 API 一 JSON；模块、API 原名与 API 文档一致。 | 跨模块、替换点号或合并多个 API。 |
| EC 8.1 字段忠实 | 用例中的字段、类型和取值范围均可在 API 文档中找到。 | 虚构字段、类型不符或越出范围。 |
| EC 8.2 errno 忠实 | EX 中 errno 位于 API 文档错误码表；无法模拟者放入 `_未覆盖错误码说明`。 | 虚构 errno 或套用其他 API 的 errno。 |
| EC 8.3 平台分支忠实 | `platformBranches` 和 TC 平台限定逐字符合文档的三端支持/差异说明。 | 把不支持平台写为支持、漏掉明示差异或限定矛盾。 |

### 有效性

| ID | 通过条件 | 失败信号 |
| --- | --- | --- |
| EC 9 描述可执行 | TC 有动词短语和具体值；平台、次数、数据量在适用时写明；EX 有 success/fail 和关键断言。 | “测试 X”“大数据”等空泛 TC，或 EX 没有可断言结果。 |
| EC 10 描述具体度 | 同时满足具体 TC、明确 EX 关键断言，以及适用的限定条件。 | 上述任一部分缺失。 |
| EC 11 环境可达 | 用例平台限定与文档支持平台一致。 | 指向不支持平台，或遗漏文档要求的平台限定。 |
| EC 12 可验证性 | 有副作用的用例说明清理；并发用例断言全部子调用成功并清理；流程用例断言中间状态。 | 缺少相应清理或关键断言。 |

### 可维护性

| ID | 通过条件 | 失败信号 |
| --- | --- | --- |
| EC 13 内容去重 | 同一 JSON 中不存在“测试场景 + 期望结果”语义重复的 TC/EX；枚举值、边界值和平台差异形成的明确差异允许保留。 | 至少一对 TC/EX 语义重复。 |

## 判定辅助规则

执行 EC 1–EC 4、EC 6、EC 9–EC 12 时，参照 [`EC 用例评审 — 设计规则速查表.md`](EC%20用例评审%20—%20设计规则速查表.md)：

- 入参类型矩阵和特殊补充用于 EC 2；
- T01–T11 触发信号用于 EC 4；
- 并发、大数据、权限等高级场景信号及错误码处理用于 EC 3；
- TC/EX 前缀、具体度和 `canIUse` 特殊结构用于 EC 1、EC 6、EC 9、EC 10；
- 有副作用、并发和流程断言要求用于 EC 12。

## 评分与结论

每次可完成的评审计算四维分数（0–100）：

| 维度 | 权重 | 评分依据 | 对应规则 |
| --- | ---: | --- | --- |
| 全面性 | 50% | 入参/出参（含嵌套）在 `schemaList` 与标准用例中的覆盖率；应命中 T01–T11 类型的实际覆盖率。 | EC 1–EC 4 |
| 正确性 | 25% | 文件、schema、模块、字段、errno 与平台分支是否忠实于文档。 | EC 5–EC 8.3 |
| 有效性 | 20% | TC/EX 的可执行性和具体度，平台可达性，副作用/并发/流程断言。 | EC 9–EC 12 |
| 可维护性 | 5% | EC 13 无重复为 100，有重复为 0。 | EC 13 |

`综合评分 = 全面性 × 50% + 正确性 × 25% + 有效性 × 20% + 可维护性 × 5%`

| 条件 | 结论 |
| --- | --- |
| 综合评分 ≥ 90，且没有 `fail` | `approved` |
| 综合评分 ≥ 70，且未满足 approved 条件 | `changes_requested` |
| 综合评分 < 70 | `rejected` |

## 报告模板

选择一种且仅一种模板。报告仅填充模板字段；没有数据写 `n_a`。

### 范围不完整

```text
[@MSI质量运营助手:MSI_OPS_BOT_ID] 无法继续：EC 描述评审需要明确目标。请提供以下任一项：
1) JSON 文件路径（msc/EC_TestCases/<module>/<api>.json）
2) 分支链接 + API 名 + 模块
```

### 阻塞

```text
[@MSI质量运营助手:MSI_OPS_BOT_ID] MSI EC 用例描述评审失败（阶段：<failedStage>）

[失败阶段] <当前执行状态>
[原因] <原始错误或缺失信息>
```

### approved

```markdown
[@MSI质量运营助手:MSI_OPS_BOT_ID] 本次 MSI EC 用例描述评审**通过**，请通知原始触发人继续后续流程。

[评审类型] ec-test-case-spec
[评审结论] approved
[需求来源] <requirementSource>
[分支链接] <branchUrl>
[综合评分] <0-100>
[原始触发人] [@<triggerUser 中文名>:<triggerUserId>]

| 维度 | 权重 | 分数 |
| --- | ---: | ---: |
| 全面性 | 50% | <分数> |
| 正确性 | 25% | <分数> |
| 有效性 | 20% | <分数> |
| 可维护性 | 5% | <分数> |

| EC ID | 结论 | 证据 |
| --- | --- | --- |
| EC 1–EC 13 | pass / n_a | <证据或 n_a> |
```

### changes_requested / rejected

```markdown
[@MSI质量运营助手:MSI_OPS_BOT_ID] 本次 MSI EC 用例描述评审**未通过**，请按扣分项和待修改项修改后再次提交审核。

[评审类型] ec-test-case-spec
[评审结论] <changes_requested 或 rejected>
[需求来源] <requirementSource>
[分支链接] <branchUrl>
[综合评分] <0-100>
[原始触发人] [@<triggerUser 中文名>:<triggerUserId>]

| 维度 | 权重 | 分数 | 扣分项（EC ID + 原因） | 待修改（JSON 路径 + 修改方向） |
| --- | ---: | ---: | --- | --- |
| 全面性 | 50% | <分数> | <内容或 无> | <内容或 无> |
| 正确性 | 25% | <分数> | <内容或 无> | <内容或 无> |
| 有效性 | 20% | <分数> | <内容或 无> | <内容或 无> |
| 可维护性 | 5% | <分数> | <内容或 无> | <内容或 无> |

| EC ID | 结论 | 证据 |
| --- | --- | --- |
| <每个 fail / n_a> | fail / n_a | <相对路径 : JSON 路径 | 节选> |
```
