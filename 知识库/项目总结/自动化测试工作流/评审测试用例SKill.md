## §3 评审维度 A:完整性(Completeness)

  

| ID | 规则 | pass | fail |
|--------|------------|--------------------------------|-----------|
| **C 1** | 文件位置正确 | 文件落在 `msc/AI_TestCases/` 下 | 不在该目录 |
| **C 2** | 必备字段齐全 | 每个用例对象至少含 `apiName` / `desc` / `run` 三字段 | 任一缺失 |
| **C 3** | 覆盖正常路径 | 至少 1 条非 `canIUse` 非异常的"基础调用"用例 | 全是 canIUse / 异常 / 文件为空 |
| **C 4** | 覆盖 canIUse | 1 条针对该 API 的 `canIUse` (断言 `canIUse(apiName) === true`) | 无任何 canIUse |
| **C 5** | 覆盖异常 / 边界 / 兼容性 | 至少 1 条入参非法 / 缺参 / 越界异常用例 / 兼容性 | 无任何异常或边界 |
| **C 6** | JS 完整覆盖 EC JSON | 协议含 `[EC描述路径列表]` 时启用:期望 JS 实际 case key 集合覆盖 EC case key 集合(EC JSON 构造) | JS 实际用例未覆盖 EC Case |
| **C 7** | 关键断言齐全 | 每条用例 `run` 内含**关键断言**;异常断 `errno` 或 `errMsg`;流程断中间状态;并发 / 大数据断全部子调用成功 + 资源清理 | 断言宽松(只 `expect(res).not.to.be.null`)或缺关键字段 |

  

### §3.1 C 6 评审执行细节

  

依赖协议透传的 `[EC描述路径列表]` 步骤:

  

1. **读 EC JSON**: `git show <分支名>:<json路径>`

2. **构造期望 case key 集合**:

- `标准用例.canIUse` → `<api>_canIUse`

- `标准用例.入参.<字段>` / `标准用例.出参.<字段>` / `场景测试.<类型>` 下每条 TC × EX → `<api>_<scenario>_<expected>`

3. **读 JS 实际 case key 集合**:从候选 `.js` 读 `cases[].key`

4. **比对差集**: `差集 = 实际集合 - 期望集合`

- 实际集合 **覆盖** 期望集合 → C 6 pass

- 实际集合 **未覆盖** 期望集合 → C 6 fail,证据列每条缺失 case key + 对应 EC JSON 路径 + TC 描述

1. **协议缺 `[EC描述路径列表]` 时**: `C6 = n_a` (不计入分母), `[整体结论]` 备注"本次派活未透传 EC JSON 路径,C 6 跳过"

  

### §3.2 C 7 评审执行细节

  

确保用例不只"形式上跑过"而是"真正验证业务逻辑"。步骤:

  

1. **扫每条 cases 数组项**,读 `run` 函数体(或 `assert` / `expect`)

2. **按用例类型判定关键断言要求**:

- **canIUse 类**(`<api>_canIUse`):必含 `canIUse(...) === true`;有 `platformBranches` 还需含平台分支断言

- **异常**(case 名含 `error` / `invalid` / `null` / `undefined` / `boundary`):必断 `errno` 或 `errMsg` 关键字段

- **流程**(case 名含 `setStorage→getStorage` / `connect→send→close` 串联):必断中间状态(如 setStorage 后调 getStorage 拿到的值 === 原值)

- **并发**(case 名含 `并发` / `concurrent`):必断 `Promise.all` 返回数组全部 `success` + 资源清理(如 SocketTask 全部 closed)

- **大数据**(case 名含 `1MB` / `5MB` / `10000`):必断成功 + 清理副作用

- **正常路径**:至少断 `res.success === true` 或返回对象关键字段非空

3. **fail**:用例存在但断言宽松(仅 `expect(res).not.to.be.null` / `expect(...).to.exist`)或缺关键字段断言,证据列文件 + 行号 + 用例 key + 当前断言代码

4. **n_a**:C 6 已判 fail 时,C 7 跳过(等 C 6 修完再评);否则一律 evaluate

## §5 报告输出格式

  

回复模板**两类,按 §5.0 综合评分阈值二选一**:

  

| 综合评分 | 结论 | 走哪个模板 |

|---|---|---|

| ≥ 90 且无 fail | `approved` | §5.1 |

| 70 ≤ 评分 < 90 或有 fail 但 ≥ 70 | `changes_requested` | §5.2 |

| < 70 | `rejected` | §5.2 |

  

### §5.0 评分体系(决定走哪个模板)

  

每次评审必算 **4 维加权综合评分**(0-100,LLM 主观打分):

  

| 维度        | 权重  | 打分口径                                                                            | 对应 C / T                     |
| --------- | --- | ------------------------------------------------------------------------------- | ---------------------------- |
| **功能覆盖率** | 30% | JS 用例完全覆盖 EC 用例得满分,未覆盖按覆盖百分比打分                                                  | C 6                          |
| **流程覆盖率** | 30% | 11 类场景(T 01-T 11)中**该 API 应命中**的清单与实际命中数比例                                      | T 01-T 11(含 C 3 / C 4 / C 5) |
| **用例质量**  | 35% | desc 具体度(动词短语 + 具体值)+ `run` 内断言能区分 success/fail + 含关键字段(canIUse / errno / 中间状态) | C 2 + C 7                    |
| **用例重复率** | 5%  | desc 语义重复(只看 desc 文本表达的测试场景,允许枚举值 / 边界值 / 平台差异等显式区分),有重复用例得 0 分                 | —                            |

  

**综合评分** = `(功能覆盖率 × 30% + 流程覆盖率 × 30% + 用例质量 × 35% + 用例重复率 × 5%)`

  

**说明**:

- C 1(文件位置正确)是前置门槛——fail 直接走 §6 失败处理(文件不在 `msc/AI_TestCases/` 下,无法评审),不进入评分

- 用例重复率**只看 desc 语义**,禁字符串相似度判定
