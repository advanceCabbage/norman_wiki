# 一、目标

深度融合 AI 大模型与既有自动化基建，构筑 **MSI 端到端智能化质量保障系统**：

1. **全链路贯穿**：打通 “EC 用例生成 → 可执行用例生成 → 代码与用例一致性校验 → APP 构建 → 多容器并发自动化执行 → 测试报告上传” 全生命周期
    
2. **多端覆盖**：全面适配 Android、iOS、HarmonyOS 三端生态
    
3. **多容器保障**：覆盖美团/点评双端 App 及其 MSC、MRN、Mach 业务容器

### 3.2.2 “Skill 能力” 与 “自动化脚本” 的职责边界

为了保障高并发执行的确定性，方案严格划分了 **大模型 skill**（负责语义理解、逻辑推理、代码生成）与 **工程自动化脚本**（负责环境拉起、代码拉取、物理执行）的边界：

1. **封装为大模型 skill 执行的功能（依赖推理）**：
    
    - 需求理解、用例逻辑编写、用例逻辑自省纠偏、代码 Diff 变更特征提取
        
2. **封装为确定性脚本执行的功能（依赖工程基建）**：
    
    - **代码及分支管理**：封装 Git 脚本，自动识别 PR 所属端（iOS / Android / Harmony）并提取 Diff
        
    - **三端多容器执行引擎**：调用 Mpium 框架执行 App 唤起与授权；动态构建 MSC / MRN / Mach 容器 Demo 环境并执行 JS 层测试用例
        
    - **数据流转**：自动将测试报告上传至 S 3 存储，并调用大象 OpenAPI 触发群通知
        

### 3.2.3 协同收敛状态机

协同过程引入最大 3 次 Loop 的硬熔断机制。审核通过则由质量运营机器人通知 RD 验收；若 3 次交互后仍未收敛，触发熔断，直接转人工介入，确保流水线不卡死

# 五、工程实现与自动化运行

### 5.1 OpenClaw 对接大象渠道

**5.1.1 通信原理与链路改造**

OpenClaw 的消息通信机制基于 `Channel` 抽象实现。因原生不支持大象渠道，方案通过自建大象 Channel 插件打通数据链路。

- **核心痛点**：大象采用 Webhook 回调机制（用户发消息 ➔ 大象服务端 ➔ 回调用户自建服务）。但 OpenClaw 运行在 CloudIDE 或本地个人电脑上，无法作为公网服务直接接收大象的回调。
    
- **解决思路**：部署中间服务 `meishi-dxopen` 注册为大象的回调地址，用于接收和暂存消息。OpenClaw 端的 Channel 插件通过长轮询（Long-polling）方式，主动向该中间服务拉取消息并回复。
    
- **完整链路**：用户（大象 App） ➔ 大象服务端 ➔ `meishi-dxopen` 服务（接收回调） ➔ OpenClaw Channel 插件（长轮询） ➔ Agent ➔ 原路返回
    

##### 5.1.2 多用户与多机器人消息隔离机制

- **私聊隔离**：单聊机器人采用独立 Session，Key 中携带用户的 `uuid` 作为唯一标识。
    
- **群聊隔离**：群聊机器人以 `群ID` 作为 Session 的部分标识；群内多人交互时，以 `群ID + 用户uuid` 作为联合唯一标识，确保多人同时对话不串台。
    
- **上下文补充**：大象 Channel 在群聊场景下会默认携带最近 10 条历史消息一起发送，确保 Agent 具备足够的对话上下文。
    
- **机器人身份识别**：在身份配置文件 `IDENTITY.md` 中明确定义角色属性。规定在大象群消息上下文中，所有「发言者 = MSI 质量运营助手」的消息均视为机器人自身发言，而「审核者机器人」的发言则识别为外部意见，避免机器人混淆自身与审核者的对话身份
    

### 5.2 OpenClaw 接入 Codex 作为 Runtime Agent

**5.2.1 背景与替换动因**

OpenClaw 默认由 `PI Agent` 驱动，其在框架分层与记忆机制上设计优良，但在代码搜索与编写场景下，Codex 表现更优。鉴于 MSI 端到端智能测试中“生成 EC 用例、生成可执行用例、PR 与用例差异比对”等环节强依赖代码读写，方案采用 Codex 替换底层 Runtime Agent。关于 `PI Agent` 介绍也可参考 [MSI AI智能搜索方案](https://km.sankuai.com/collabpage/2751950834)

##### 5.2.2 可插拔的 Runtime Harness 机制

OpenClaw 的 Agent Runtime 并非硬编码在核心流程中，而是抽象为统一的 `Harness / Runtime` 接口层

- **非旁路外挂**：Agent 推理、工具调用与上下文推进均由统一接口承接
    
- **平滑接入**：Codex 作为一种具体的 Runtime 实现，直接嵌入 OpenClaw 已有的执行框架中，无需重构核心流水线
    

##### 5.2.3 接入 Codex 后，哪些能力保留在 OpenClaw，哪些交给 Codex

Codex 接管的主要是 runtime 执行层，而不是产品状态层。也就是说，接入 Codex 之后，OpenClaw 原有的 session、身份信息、skill 和记忆机制并不会被整体替换掉。可以简单理解为三层分工：

- **OpenClaw 持有产品层状态**：包括 session、transcript、auth profile、skills、memory/context policy
    
- **Harness 负责 runtime 抽象层**：负责决定本次 agent 运行交给哪个 runtime 执行
    
- **Codex 接管执行层**：负责模型回合推进、thread/turn 执行、工具调用链路，以及部分上下文压缩生命周期
    

所以，Codex 能够在 OpenClaw 中较为丝滑地替换默认 runtime agent，根本原因有两个：

- OpenClaw 先天就把 agent runtime 设计成可插拔抽象
    
- OpenClaw 的产品层能力和 runtime 执行层是分开的
    

因此，在代码工程场景下，团队既可以保留 OpenClaw 原有的 session、身份、skills 和记忆体系，又可以利用 Codex 在 agentic coding 方面更强的执行能力。这就是它能平滑接入、并且具备实际价值的核心原因

### 5.3 测试用例与代码一致性评审原理
![[23c7ffe6-e997-4f0d-ab18-6c520160ba26.png]]
##### 5.3.1 上下文自动化构建机制（Pipeline 准入）

接收 PR 信号后，前置脚本 `prepare-context.sh` 静态提取并对齐源码与用例，构建统一上下文：

- **输入参数**：承接 `PR_URL`、`CASE_BRANCH_URL`（用例分支）及 `TRIGGER_USER_JSON`（触发人凭证）
    
- **平台路由**：解析 PR 提取 `org/repo`，检索 `platform-repo-mapping.md` 自动推断目标平台（三端）与基线分支
    
- **源码 Diff 提取**：拉取单端代码仓并 Fetch 最新 MR Ref；精准过滤语言后缀（如 `.swift/.m/.kt/.java/.ets/.ts/.js` 等），生成 `code.diff`
    
- **增量用例提取**：比对 `msi-auto-test` 仓分支与 `master` 基线差异，静态提取改动的 `msc/AI_TestCases//*.js` 文件
    
- **归一化产物**：将元数据、Diff 路径及用例集封装为标准 `context.json`，作为后续 AI 评审的唯一输入源
    

##### 5.3.2 智能校准推理与双向决策链路

- **LLM 差异判定**：模型严格对比 `code.diff` 与已有用例，不脑补、不输出修复建议，仅识别两类客观偏差：
    
    - **覆盖性（场景 1）**：代码中新增了用例未覆盖的错误码（`errno`）、新入参或新条件分支
        
    - **一致性（场景 2）**：已有用例的期望结果与新代码行为冲突（如错误码改值、出参结构删改、分支移除）
        
- **群报告与双向决策**：流水线通过 `render-report.sh` 渲染差异报告并推送到大象群，由 **RD 进行双向对齐决策**：
    
    - **A) 以代码为准**：触发 `generate-test-case` 技能，全自动增补、修改或删除用例完成对齐
        
    - **B) 以用例为准**：判定 PR 不通过并阻断合入，通知 RD 回归修正代码
        

### 5.4 自动执行测试用例脚本

##### 5.4.1 自动化执行流程与核心依赖

打通原生自动化框架与业务容器，实现测试用例端到端的全自动无人值守执行

- **核心依赖物料**：
    
    - **业务容器**：提前构建完成的 MSC、Mach、MRN 容器 Demo
        
    - **测试资产**：基于 JavaScript（JS）编写的标准化测试用例
        
    - **驱动引擎**：`mpium` 自动化测试框架
        
- **物理执行顺序**：
    
    1. **启动容器并获取链接**：拉起 MSC、Mach、MRN 容器 Demo，动态获取各业务页面的启动 URL
        
    2. **App 唤起与前置授权**：调用 `mpium` 框架执行 App 安装与启动，自动模拟点击完成一键登录及系统级权限授权
        
    3. **容器注入与模拟交互**：由 `mpium` 唤起对应的容器目标页面，借助框架的模拟点击能力，驱动页面内的 JS 测试用例物理跑通
        
    4. **高并发批量调度**：测试调度引擎按模块分发，对多个测试用例执行高并发批量跑测，最大化压缩整体耗时
        

##### **5.4.2. 测试报告核心要素与产物交付**

用例执行完毕后，系统将多维数据进行聚合，产出具备高可读性的效能报告：

- **报告核心内容**：
    
    - **执行状态统计**：明晰展示测试用例的成功、失败总量及分布明细
        
    - **多版本一致性校验**：自动比对多个不同迭代版本间测试用例的真实输出值一致性，严防核心业务逻辑发生非预期变动或逻辑回归
        
- **数据流转与交付**：测试报告生成后自动上传至 S 3 存储。随后，系统调用大象 OpenAPI 触发群消息通知，将结构化报告卡片推送给研发（RD）进行最终验收
