# EC 用例描述评审

以**证据链**完成 MSI EC JSON 用例描述的只读静态评审：每一条结论都能回溯到 API 文档、目标 JSON 和 EC 规则；稳定评审过程，不要求不同 API 得到相同结论。

本 Skill 只评 `msc/EC_TestCases/<module>/<apiName>.json` 的 JSON 描述层（EC 1–EC 13），不评 `msc/AI_TestCases/` 下的可执行 JS 用例。

## 必要参考资料

- 执行 EC 规则、计算评分或生成报告时，读取 [`评审EC用例 — 规则与报告模板.md`](评审EC用例%20—%20规则与报告模板.md)。它是 EC 1–EC 13、评分和输出模板的唯一事实来源。
- 判断字段类型、用例场景和 `canIUse` 结构时，读取 [`EC 用例评审 — 设计规则速查表.md`](EC%20用例评审%20—%20设计规则速查表.md)。

## 评审步骤

### 1. 确认派活与评审范围

仅将同时满足下列条件的消息视为本 Skill 的派活：

- `[评审类型] ec-test-case-spec`；
- 发送者或显式提及可确认是 `[@MSI质量运营助手:MSI_OPS_BOT_ID]`；
- 有 `[需求来源]`、`[分支链接]`、`[API名]`、`[模块]`、`[评审轮次]`、`[评审上限]`，可选 `[变更文件]`。

锁定 `triggerUser`、`requirementSource`、`apiName`、`module`、`round`、`maxRounds` 和分支名。评审范围按以下顺序定位：`[变更文件]` 中的目标 JSON → `[模块]` + `[API名]` 推导的 JSON 路径 → 分支 diff 中的目标 JSON。

若范围不足以唯一定位一个目标文件，使用参考资料中的“范围不完整”模板一次性请求补充；若协议或分支链接无法验证，使用“阻塞”模板结束本轮。

**完成标准：** 已记录不可变的派活身份、需求来源、分支、轮次与唯一 JSON 相对路径；或已按模板发出唯一的补充/阻塞回复。

### 2. 建立证据链

从目标分支只读读取 JSON；分支读取使用 `git show <branch>:<relative-path>`，工作目录文件则直接读取。解析 JSON，并从 `requirementSource` 提取 API 名、模块、入参/出参（包含嵌套字段）、错误码、三端支持差异、范围与业务约束。

为每一个可判定结论记录：`相对路径 : JSON 路径 | 不超过 120 字的原文片段`。缺失 API 文档或 JSON 无法解析时，记录原始错误并使用阻塞模板结束本轮。

**完成标准：** JSON 已成功解析，或已带原始错误终止；可用于逐条规则判断的文档事实和 JSON 证据已齐备。

### 3. 完整执行 EC 1–EC 13

读取“规则与报告模板”中的 EC 规则，对所有适用规则逐条判定 `pass`、`fail` 或 `n_a`。先检查文件位置与 JSON 结构，再检查全面性、正确性、有效性和可维护性。

每个 `fail` 都关联 EC ID、证据链和可直接修改的 JSON 路径；没有证据时，结论为 `n_a` 或阻塞，而不是推断为 `fail`。三端（iOS、Android、HarmonyOS）差异必须随同文档一起检查。

**完成标准：** EC 1–EC 13 均已有判定；每个 `fail` 有 EC ID、证据和修复方向；每个 `n_a` 说明无法判定的输入缺口。

### 4. 计算结论并生成报告

按参考资料的四维评分口径计算综合评分，再由“评分—结论”表选择 `approved`、`changes_requested` 或 `rejected`。将所有评分、EC 判定和证据填入对应的唯一报告模板。

报告只输出所选模板要求的字段和表格。`approved` 使用通过模板；`changes_requested` 与 `rejected` 使用未通过模板；阻塞情况使用阻塞模板。

**完成标准：** 结论与评分阈值一致，报告模板完整，且每一个失败项可追溯到证据链。

### 5. 提交前自检

逐项核对：评审对象是 JSON 描述而非 JS；分支链接已原样透传；`triggerUser` 未变化；所有 `fail` 映射 EC 规则并附证据；四维分数、权重和综合分计算正确；报告未包含模板外内容。

**完成标准：** 全部自检项通过后才发送报告；任一项不成立时回到对应步骤修正。
