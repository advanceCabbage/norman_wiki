# Claude Code 不用 RAG，如何完成代码检索？

很多人会默认把“代码检索”与 RAG 绑定：切片、向量化、召回，再把相关代码交给模型。但 Claude Code 走了另一条路——它不预处理、不建索引、不算向量，而是在需要时直接到代码现场检索。

它依靠的是 **Grep、Glob、Read** 三件套，并在复杂任务中派出子 Agent 探索。核心并不只是工具替换，而是把“找什么、何时找、下一步怎么找”的决策权交还给 LLM。

## 一、为什么代码检索不一定适合 RAG

RAG 的隐含前提是：LLM 还不够强，需要工程系统提前替它准备材料。于是有了 chunking、embedding 与向量召回。

但在代码场景里，这条路径有几个天然痛点：

1. **冷启动**：Grep 可以毫秒级直接搜索；RAG 则需要先建索引，通常是分钟级冷启动。
2. **实时性**：Grep 每次读取磁盘上的最新文件；代码一旦修改，RAG 索引可能滞后，还需要重建。
3. **精确性**：Grep 是确定性的字符或正则匹配。例如搜索 `getUserById`，结果就是该符号；向量检索是近似匹配，可能混入大量相似函数。
4. **Token 与成本**：Grep 配合 Read 按需读取，模型只看需要的几行；RAG 要为整个代码库做 embedding，并承担存储和计算成本。
5. **可解释性**：Grep 的每一步搜索都可见、可审计；RAG 的 Top-K 召回相对黑盒，出问题时不易排查。
6. **决策权**：Grep 让 LLM 在每轮自己决定搜什么、读什么，并通过多轮迭代逐步逼近答案；RAG 往往是一次性准备材料，模型只能基于已召回的内容作答。

换句话说，Claude Code 的假设是：模型已经能做检索决策，工程系统更应提供可靠工具，而不是替模型决定哪些材料相关。

## 二、代码检索三件套：Glob、Grep、Read

Claude Code 每次需要代码时再去现场查。三件工具分别覆盖“找文件、找内容、读上下文”三个动作。

### 2.1 Glob：先按路径缩小范围

**Glob** 按文件路径或名称模式查找文件，输出匹配的文件路径列表，对应常见的 `find` 场景。

它有两个关键约束：

- 结果按修改时间倒序排列，最近修改的文件优先出现。这是一条朴素的启发式规则：最近改动的内容通常更可能与当前任务相关。
- 结果最多返回 100 个文件，避免搜索结果撑满上下文。需要更多时，模型应收紧 pattern 后再次搜索。

### 2.2 Grep：按内容精确定位

**Grep** 按文件内容的正则表达式搜索，底层使用 Rust 实现的 ripgrep。它支持三种输出：`content`（匹配行）、`files_with_matches`（命中文件列表，默认）与 `count`（命中数量）。

它的优势包括：

- **权限统一管控**：将 Grep 独立封装为工具，避免模型通过 Bash 直接调用 `grep` 或 `rg`，从而收紧权限并约束使用方式。
- **输出格式可控**：工具可约定结构化输出，例如携带行号；Bash 的原始文本输出则较难统一。
- **性能可靠**：ripgrep 支持多线程、自动遵从 `.gitignore`，通常比传统 `grep` 更快。
- **结果受限**：一次最多输出 250 项，可通过分页继续读取，防止上下文爆炸。
- **适合开放式搜索**：当搜索需要多轮、探索性推进时，工具定义会引导模型改用 Agent 工具。

工具定义的核心约束是：

```text
A powerful search tool built on ripgrep
- ALWAYS use Grep for search tasks. NEVER invoke `grep` or `rg` as a Bash command.
- Output modes: "content" shows matching lines, "files_with_matches" shows only file paths (default), and "count" shows match counts.
- Use Agent tool for open-ended searches requiring multiple rounds.
```

### 2.3 Read：只读取真正需要的内容

**Read** 读取指定绝对路径的文件内容，对应 `cat` 或编辑器跳转查看文件的能力。

默认最多读取文件开头的 2,000 行；如果已经知道目标位置，应只读取对应片段。它始终读取最新内容，不做缓存、索引或预处理。

这套设计的核心是：**按需读取，不要贪心。**

## 三、三件套不够时，让 Explore 子 Agent 去探索

大项目中的开放式检索可能要调用十几个工具，并产生大量中间结果。若这些内容都进入主 Agent 的上下文，会造成严重污染。

此时，主 Agent 可以派出 Explore 子 Agent。典型流程如下：

1. 主 Agent 通过 Agent 工具派发一个探索任务。
2. 子 Agent 获得精简工具池，通常以只读工具为主：Grep、Glob、Read，以及只读 Bash 命令。
3. 子 Agent 在自己的上下文中进行多轮检索和推理。
4. 子 Agent 完成后，只把最终结论返回给主 Agent。

这形成了三层分工：

- **底层**：Grep、Glob、Read，处理简单、定向的检索。
- **中层**：Explore 子 Agent，负责开放式探索，并隔离中间上下文。
- **上层**：主 Agent 编排整体任务。

## 四、LSP 是语义导航的补充，而不是替代品

Claude Code 默认没有安装 LSP 插件，需要手动安装。以 TypeScript 为例：先通过 `/plugin` 安装 `typescript-lsp` 插件，再全局安装 `typescript-language-server` 与 `typescript`；之后 Claude Code 会在本机启动语言服务器进程。

LSP 更适合“已知符号、需要精确定位”的任务，例如：

- 已知函数、类型或变量名，要查定义、全部引用、调用者或被调用者。
- 获取类型信息或 hover 文档。
- 跨文件进行语义级导航，并区分同名但不同作用域的符号。

Grep 和 Glob 则仍适用于文本或文件模式匹配：探索某项功能在哪里被使用、搜索跨语言或非代码文件、按路径寻找测试文件，以及查注释、字符串、日志和文档。

二者常常配合：先用 Grep 找到大致位置，再用 LSP 根据精确的 line/character 位置执行 `findReferences`、`goToDefinition` 等语义导航，而不是从零开始只依赖 LSP。

## 五、结语：把检索决策交回模型

Claude Code 不依赖 RAG 完成代码检索，并非否定检索工程本身，而是选择让模型在循环中主动决策：先搜什么、读哪些文件、何时扩大或缩小范围，以及何时委派子 Agent。

工具负责提供确定、实时、受权限约束的能力；模型负责根据现场结果不断迭代。这正是它的代码检索机制背后的设计哲学：**不替模型做决定。**
