## 二、写好 skill 的要点
#### 2.1 调用方式
- **模型调用**：保留 description，模型可以自主发现并触发它。代价是 descripdescription 会在每轮对话都占据上下文，此时 description 是面向模型的触发词清单
- **用户调用**：设置 disable-model-invocation：true，模型看不到 description。仅用户输入 skill 名称才能调用。代价是增加用户认知负担，人必须记住它的存在和用途，此时 descriptiodescription 是面向人的一行摘要
#### 2.2 编写 description
- 精简：description 内容会增加上下文负担，因此需要精简
- 把 Skill 的**主导词**（leading word）放在前面；description 的首要任务是触发调用 ，只保留触发条件且每个分支只保留一个触发条件
**示例**：description: 用例评审：评审 `TestCases` 中的 JSON 用例描述。用于功能用例评审请求或 `[评审类型] test-case-spec` 派活协议
> **用例评审**作为主导词，以及说明两个触发条件分支

#### 2.3 主导词
主导词是模型预训练中已有的、简洁但有行为含义的概念，例如 _lesson_、_fog of war_、_tracer bullets_。它让模型用一个词锚定一整片行为规则；重复出现时，定义分布在全文中逐渐形成。

它两次提高可预测性：
- 在正文中，它锚定**执行**：每次遇到该词，模型都会联想到相同的做法。
- 在 description 中，它锚定**调用**：当该词也出现在用户提示、文档和代码中，模型更容易将它关联到此 Skill。

应持续寻找可用主导词合并重复表述。例如：
- “快速、确定、低开销”可压缩为 _tight_（紧凑的循环）。
- “一个你确信成立的循环”可压缩为 _red_：测试对 bug 失败就是 red，否则不是。

收益不只是更少 token，也是更锋利、更稳定的行为锚点。优先使用模型已经理解的词；自造词必须额外花篇幅定义，无法借用模型既有知识

#### 2.4 信息层级
skill 的内容可以由**步骤**（steps）和**参考资料**（reference）任意组合：可以全是步骤、全是参考或两者兼有
- **skill 内步骤**：**步骤需要是有序的且每一步都有完成标准要求**，目的是让大模型能知道当前步骤完成的标准
- **skill 内参考资料**：需要按需查阅的定义、规则与事实
严格的步骤要求及完成标准，可以避免大模型过早的完成任务；参考资料按照文件分类并在 Skill.md 中设置索引关系，可以让顶层保持清晰，大模型通过判断分支加载对应的关联内容，实现渐进披露，减少上下文负担

#### 2.5 何时拆分 skill
- **按调用拆分**：当某一能力有独立的主导词、应被模型单独触发，或必须供其他 Skill 调用时，拆成模型调用型 Skill。代价是新增一个始终加载的 description。
- **按顺序拆分**：当后续步骤（**完成后的步骤 / post-completion steps**）会诱使模型草草结束当前步骤时，把步骤序列切开。让后续步骤暂时不在眼前，能促使模型对当前任务做更多前置功

#### 2.6 常见的失效模式
- **过早完成（premature completion）**：当前步骤尚未完成，就把注意力转向“结束”。**先把完成标准写得更明确、可检查**；只有标准天生无法明确且确实观察到仓促行为时，才通过拆分来隐藏后续步骤。
- **重复（duplication）**：**同一含义出现在多个地方**。它增加维护和 token 成本，并不当地抬高该信息的重要性。与有意重复“主导词”不同，重复的是完整含义而非单个行为锚点。
- **沉积（sediment）**：因为“添加安全、删除有风险”而累积的过时内容。没有定期精简，Skill 默认会变成这样。
- **蔓延（sprawl）**：即使每一行都有效且不重复，文件仍然过长。它降低可读性和可维护性，并浪费 token。**用信息层级下沉参考资料，并按分支或步骤序列拆分**。
- **无效指令（no-op）**：**模型默认就会做的事，被再说一遍却没有改变行为**。例如模型本已“还算仔细”，再写“要仔细”几乎无效；应改为更有力度、能改变行为的词，如 _relentless_。是否为无效指令取决于模型的实际默认行为，应通过运行验证而非争论。
- **否定式引导（negation）**：告诉模型“不要做 X”会把 X 放进注意力中心，反而更容易发生。尽量正向描述目标行为，**例如用“注释保持一行”替代“不要写冗长注释”**。只有不能正向表达的硬性安全边界才保留禁止语，并同时给出应采取的正向行为
## 三、实践清单
- 先定义这份 Skill 要稳定的**过程**，而不是要求稳定的输出
- 判断谁需要发现它：模型/其他 Skill 需要则模型调用型；仅人手动使用则用户调用型
- 为模型调用型 description 写最少且互不重复的触发分支，并以前置主导词开头
- 将每个动作写成带可检查、尽量穷尽完成标准的步骤
- 所有分支都要用的信息留在正文；仅部分分支需要的参考资料移到有明确触发条件的链接文件。
- 将定义、规则和例外共置；让每个含义有唯一事实来源。
- 删除不相关、过时、重复和不改变默认行为的句子；把否定式提醒改成正向目标。
- 若仍然过长，先下沉参考资料；若当前步骤总被后续任务催促，再按步骤序列拆分。
