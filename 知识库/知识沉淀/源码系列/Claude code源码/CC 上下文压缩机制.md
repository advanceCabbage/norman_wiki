## 一、核心结论
Claude code 的上下文压缩，本质是一次上下文重建。它不是删除会话记录，也不是简单截断最早消息，完整的文字记录（transcript）仍会持久化；被压缩掉的是后续主模型请求中不再自动携带的旧消息
## 二、触发压缩的四种类型或时机
###### 2.1 自动压缩
**主 query loop 每次准备请求主模型前，都会检查 token 数量，达到自动阈值后，先压缩，再发送主模型请求**。自动压缩可以配置关闭和配置自动压缩窗口；关闭后，接近极限时系统会阻止继续请求，保留空间让用户手动执行 `/compact`
- 有效上下文窗口大于 800 K，则预留 50 K，阈值约为 93%
- 有效上下文窗口小于 400 K，则预留 13 K，阈值约为 96%

###### 2.2 手动 /compact
手动压缩不必等待 token 超限，并且可以提供自定义摘要要求
举例：
```
/compact 请重点保留测试结果和代码改动
```

###### 2.3 `prompt_too_long` 的恢复压缩
部分 feature-gated 路径中，主模型已经返回“上下文过长”错误时，Claude Code 会暂缓暴露错误，尝试 reactive compact，然后在同一 query loop 中重试请求

###### 2.4 局部压缩
用户通过历史消息选择器，可以只压缩某个位置之前或之后的消息，这是用户控制上下文结构的高级入口，不等同于普通全量自动压缩
**操作方法**：连续按两次 Esc ，调出历史消息记录，选中用户消息 U 及结束消息，进行局部压缩

## 三、压缩相关
###### 3.1 压缩时同步还是异步
从代码执行方式看，压缩是 `async/await` 的网络调用；从 Agent loop 行为看，它是**同步阻塞的前置步骤**

###### 3.2 哪些内容参与压缩
- 最近一次的压缩总结（压缩总结前的用户消息不参与）
- 用户消息 & 助手消息（最近一次压缩总结后的消息）
- 工具调用 & 工具执行结果
压缩前还会经过一些上下文瘦身步骤，例如 tool result budget、snip、microcompact；因此摘要看到的是处理后的 model-facing messages，不保证是磁盘 transcript 的完整原文
- **tool result budget**：超限不是截断或丢弃，**首先**选择体积最大的工具结果，**其次**将完整内容写入文件，**然后**发送给大模型工具结果文件路径和一小段预览（从内容开头截取约 2000 字符内容，寻找换行符进行截取，实在找不到就直接截取），**最后**大模型如需细节可以再调用 Read 读取该文件。**两个触发时机**：
	- 单个工具结果超出 50K 字符
	- 同一个 API User 的多个工具结果聚合超过 200 K 时
- **snip**：移出历史会话，但需要由用户主动触发。**触发时机**：
	- **用户显示触发/force-snip**，snip把执行命令前的全部历史移出下一次及后续模型请求，边界后的消息保留
	- **出现 30 条消息阈值时**：系统会给大模型输入一段提示词，让大模型考虑 snip 掉一些历史消息。大模型可以调用工具 snip 裁剪掉一些历史久远的上下文。后续在进行 autoCompact 时，以及被 snip 裁剪掉的内容不再参与自动压缩
- **microcompact**：清空旧工具观察，而不是压缩整段对话。**两个触发时机**：
	- **按空闲时间清理**：当两次交互之间间隔超过配置阈值（默认 60 分钟），通常意味着大模型 API 端的提示词缓存已失效。**具体做法：把「可重新获取」的工具结果（Read / Bash / Grep / Glob / WebSearch / Edit / Write）清空，只保留最近 5 个**。子 agent 输出、Task 状态这类「不可重复」的结果绝不裁剪
	- **缓存型 microcompact**：本地消息不改写，而是通过 `cache_edits` 告诉服务端删掉提示词缓存中旧工具结果对应的部分。这样保留本地 transcript 和消息结构，同时缩减服务端缓存占用
tool result budget、snip、microcompact 的目标是：**在上下文真正超限、不得不做更昂贵的 autocompact 之前，先清掉历史中最常见、最占空间、通常又最容易重新读取的工具输入**

###### 3.3 压缩后会重新注入哪些信息
以下内容会在压缩后重新注入上下文

| 信息                  | 恢复策略                           |
| ------------------- | ------------------------------ |
| 最近读取文件              | 最多 5 个；每个最多 5 K token；总预算 50 K |
| Plan 文件             | 重新作为 attachment 注入             |
| Plan Mode           | 重新注入完整模式指令                     |
| 已调用 Skill           | 按最近调用优先，每个最多 5 K，总预算 25 K      |
| 异步 Agent            | 恢复未完成或未取回结果的任务状态               |
| 延迟工具状态              | 恢复已发现的 deferred tools 信息       |
| MCP / Agent listing | 重新发送当前 delta                   |
| Hooks               | 再执行 SessionStart hooks，恢复其上下文  |
| 压缩摘要                | 注入压缩摘要内容                       |
| System prompt       |                                |

###### 3.4 摘要用什么模型？
使用当前对话的同一个模型（也就是主 agent 模型）
- **摘要质量要保证**。便宜的小模型生成的摘要丢东西多，下一轮 agent 就接不上
- **Prompt Cache 复用**。prompt cache 是大模型 API 的一个能力：如果你这次请求和上次请求的开头部分（比如 system prompt、工具描述）是一样的，这部分会被缓存住，下次只算一次钱、推理也更快。Claude Code 的主对话本来就在用 prompt cache，压缩用同一个模型，能复用 system prompt 那部分的 cache，省下来的钱比换小模型省的还多

## 四、参与上下文压缩的 五层设计
#### 4.1 大结果存磁盘
**tool result budget**：超限不是截断或丢弃，**首先**选择体积最大的工具结果，**其次**将完整内容写入文件，**然后**发送给大模型工具结果文件路径和一小段预览（从内容开头截取约 2000 字符内容，寻找换行符进行截取，实在找不到就直接截取），**最后**大模型如需细节可以再调用 Read 读取该文件。**两个触发时机**：
- 单个工具结果超出 50 K 字符
- 同一个 API User 的多个工具结果聚合超过 200 K 时
#### 4.2 Snip 砍掉远古消息
**snip**：移出历史会话，但需要由用户主动触发。**触发时机**：
- **用户显示触发/force-snip**，snip 把执行命令前的全部历史移出下一次及后续模型请求，边界后的消息保留
- **出现 30 条消息阈值时**：系统会给大模型输入一段提示词，让大模型考虑 snip 掉一些历史消息。大模型可以调用工具 snip 裁剪掉一些历史久远的上下文。后续在进行 autoCompact 时，以及被 snip 裁剪掉的内容不再参与自动压缩
#### 4.3 Micro-Compact 时间衰减
**microcompact**：清空旧工具观察，而不是压缩整段对话。**两个触发时机**：
- **按空闲时间清理**：当两次交互之间间隔超过配置阈值（默认 60 分钟），通常意味着大模型 API 端的提示词缓存已失效。**具体做法：把「可重新获取」的工具结果（Read / Bash / Grep / Glob / WebSearch / Edit / Write）清空，只保留最近 5 个**。子 agent 输出、Task 状态这类「不可重复」的结果绝不裁剪
- **缓存型 microcompact**：本地消息不改写，而是通过 `cache_edits` 告诉服务端删掉提示词缓存中旧工具结果对应的部分。这样保留本地 transcript 和消息结构，同时缩减服务端缓存占用
#### 4.4 Context Collapse 读时投影
一种在发送模型请求前临时构造“模型可见上下文”的机制：它保留 REPL 中的完整原始会话，但将早期历史替换为已提交的折叠摘要、保留近期原文和关键上下文，从而减少本轮 API 输入 token，同时便于复用稳定的投影结果和 Prompt Cache。

设计上，它与传统的预防式 Auto-Compact 互斥：当 Context Collapse 启用后，Auto-Compact 不再主动把整段历史压成单一摘要，以避免两套策略在约 90% 至 95% 上下文区间竞争；但若 API 仍返回 `prompt_too_long`，系统会优先尝试 drain 已暂存的 Collapse，失败后仍可由 Reactive Compact 作为兜底**需注意，当前仓库的 Context Collapse 仅保留调用接口和设计注释，具体投影与折叠实现是 stub，尚未真正生效**。
#### 4.5 Auto-Compact 全量摘要
最重的兜底。上下文达到 ~93%（具体是「上限减 13 K」）时触发，**把整段对话全送进摘要器重写一份**，配套做文件恢复、缓存清理、消息重组。
这一层是真正意义上的「全量重写」，代价最高（要一次 API 调用）但效果也最强


## 五、压缩之后的五步骤恢复对话
- 第一步，把当前所有消息送进摘要器，生成摘要文本。
- 第二步，清空各种缓存，包括 readFileState（文件状态缓存）、loadedNestedMemoryPaths（嵌套记忆文件路径缓存）、getUserContext（用户上下文缓存）。这一步保证下一轮对话所有的状态都从干净的起点开始
- 第三步，**并发**生成各种附件：最近的文件、异步 agent 的状态、技能配置等等。注意是并发，不是顺序，这能省一些时间
- 第四步，调用前面讲过的 buildPostCompactMessages 把所有东西组装成新的消息链
- 最后一步，新的消息链替换掉旧的，对话继续

## 六、自动压缩的设计小技巧
- **摘要的开头添加说明**：摘要的开头会被包装成这样一句话，**「本会话是从之前一次因上下文耗尽而中断的对话延续过来的。以下摘要概述了之前的对话内容」**，它告诉模型一件事：**你不是从头开始的，你是接着干**
- **摘要的末尾还会带一个 transcript 文件路径**：意思是说：「如果你需要查之前对话里的某个具体代码片段或者细节，可以读这个文件」。这给了 agent 一个「翻底牌」的兜底通道，原因是历史消息仍旧保存在磁盘中
- **suppressFollowUpQuestions（禁止生成后续提问）开关**：这个开关只在 auto-compact 时打开。它的作用是：禁止摘要器在 Current Work 那部分生成「需要进一步确认」类的问题
## 七、面试可表达
**Claude Code 用的不是滑动窗口、定期摘要、向量召回那一类常见方案，而是一种叫『全量重写加分通道恢复』的工程化思路。**

第一层，触发时机。用的是绝对 token 阈值，公式是「有效上下文窗口减去 13 k 缓冲」。13 k 这个数字是基于摘要任务 p99.99 输出长度算出来的，留出来的空间刚好能安全装下摘要本身。这一层讲完，面试官就知道你不是临时抱佛脚，是真翻过源码。

第二层，取舍逻辑。反直觉的点是它不保留最近 N 条，而是把所有历史消息一刀切全部送进摘要器重写一份。关键的状态信息单独走「附件通道」恢复，比如最近读过的 5 个文件（每个最多 5 k token，总预算 50 k）、异步任务状态、当前计划文件。CLAUDE.md 这种永久指令则不进摘要，而是通过清空 getUserContext 缓存让它在下一轮自动重新加载。

第三层，摘要 prompt 设计。用 9 部分结构化清单约束输出，重点强调「所有用户消息」必须枚举不能落、「当前正在做的事」要精确到文件和函数名。摘要用的是当前对话的同一个模型，不省钱换小模型，目的是保证质量，顺便复用 prompt cache。

第四层，接续机制。压缩之后会清空文件状态缓存、并发生成附件、用 buildPostCompactMessages 把新消息链组装好。摘要外面包一句「本会话是从之前一次因上下文耗尽而中断的对话延续过来的」，让模型知道自己是接力不是从头开始。自动触发时还会打开 suppressFollowUpQuestions 开关，避免摘要里塞新问题打断当前任务。

讲完这四层，**最后用一句话收口**：「Claude Code 这套设计反映的不是『省 token』的小聪明，而是『信息分通道管理』的工程哲学。」

Claude Code 的上下文压缩不是简单删历史，而是一次上下文重建。主 loop 会在请求模型前，用 API usage 加新增消息估算上下文 token；达到模型相关阈值后，先同步 await 一个 Compact Agent 生成摘要。压缩后的上下文由 boundary、摘要和关键运行状态组成。

它通过 compact boundary 把上下文切成链：下一轮只读取最新 boundary 之后的消息，因此下一次压缩会压缩“上一份摘要加新增对话”，不会重新带入最早原文。

为了避免摘要丢失状态，Claude Code 会额外恢复近期读取文件、Plan Mode、Plan 文件、已调用 Skill、异步 Agent、MCP 和 deferred tool 状态，并重新执行 SessionStart hooks。

设计上，它把历史语义交给摘要，把精确运行状态交给 attachment，把安全和工具能力继续交给系统提示词与结构化 tools，从而在连续执行、成本、上下文长度和可靠性之间取得平衡
