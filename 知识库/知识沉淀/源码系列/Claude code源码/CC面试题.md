**1. 为什么不把所有 Skill、工具和文档一次性放进 system prompt？**

因为上下文不是越多越好。一次性塞入会带来四个问题：

- **成本高**：每轮都携带大量 token，缓存失效时尤其昂贵。
- **注意力稀释**：与当前任务无关的工具/文档会干扰模型判断。
- **过期和冲突**：大量说明可能相互矛盾，模型难以确定当前应遵守哪一条。
- **能力发现不等于内容加载**：模型只需要先知道“有什么能力”，需要时再加载细节。

所以 Claude Code 的思路是：工具通过结构化 schema 提供；Skill 首轮只给名称和描述；完整 Skill、reference、脚本说明等在需要时按需读取。这是典型的“能力目录 + 渐进式展开”。

**2. 工具描述为什么要和用户 UI 描述分开？**

因为服务对象不同。

- **给模型的工具 prompt**：重点是何时调用、输入怎么填、限制是什么、调用后能得到什么。它必须准确、稳定、偏工程化。
- **给用户的 UI description**：重点是这次具体要改什么、风险是什么、要不要批准。它应当可读、可审查，并可能展示 diff。

例如 `Edit` 工具对模型要说明 `old_string`、`new_string` 的匹配规则；对用户则应该显示“将 `a.ts` 中某段逻辑替换为另一段逻辑”及 diff。

这能避免两类常见问题：让用户看到过于模型化的指令，以及为了 UI 易读而牺牲模型调用精度。

**3. Prompt cache 如何影响系统提示词的组织方式？**

缓存要求**前缀稳定**。因此系统提示词应分层：

```
稳定、通用、长期不变的内容
-> 放在前面，作为可缓存前缀

会随会话或每轮变化的内容
-> 放在后面，例如环境、记忆、MCP 状态、动态指令
```

如果把当前时间、临时任务、实时工具列表混在最前面，前缀频繁变化，后续大段通用 prompt 也无法命中缓存。

Claude Code 的做法是把静态行为准则、工具使用原则等放在稳定区域，再设置动态边界，后面追加 memory、环境、MCP instructions 等动态 section。设计重点不是“prompt 写得多漂亮”，而是“变化最小化、顺序稳定、缓存友好”。

**4. 为什么 memory retrieval 要单独调用一个模型筛选？**

因为完整记忆库通常比当前任务大得多，不能全部注入主模型。

独立筛选模型接收：

```
当前用户问题
+ 可用记忆的文件名和摘要
```

输出少量高相关记忆文件。之后主 Agent 只接收被选中的正文。

这样做的好处：

- 减少主上下文 token 和噪音。
- 避免旧记忆、无关偏好干扰当前推理。
- 可以用小模型、短输出、结构化 JSON 完成筛选，成本较低。
- 主模型不必自己先花一轮时间“读目录、挑资料”。

关键原则是：筛选结果必须校验，例如只能返回 manifest 中存在的文件名，不能让筛选模型凭空制造记忆路径。

**5. 上下文压缩后，如何防止关键约束丢失？**

不能只让模型“随便总结一下”。要分两类状态处理：

```
不可丢失的运行时约束
-> 不依赖摘要，压缩后重新构建或重新注入

需要从历史提炼的任务状态
-> 由 compact prompt 明确要求写入摘要
```

不可丢失的约束包括系统提示词、当前工具列表、权限模式、当前配置、核心记忆、已激活 Skill 等。它们应在压缩后重新组装，而不是寄希望于摘要复述正确。

摘要 prompt 则应强制保留：

```
用户目标和限制
已完成的工作
关键文件与改动
技术决策及理由
失败尝试和用户纠正
未完成事项与下一步
```

工程上还应保留最近一段原始消息、限制摘要格式、在压缩后执行必要状态恢复。Claude Code 对已调用 Skill 有专门的 post-compact 恢复逻辑，正是为了防止“技能说明被摘要吞掉”。

**6. Prompt 要求“不要修改文件”是否足够安全？为什么？**

不够。

Prompt 只是对概率模型的行为引导，不能作为安全边界。它会受到模型误判、用户 prompt injection、上下文冲突、工具描述歧义等影响。

真正的安全边界必须位于工具执行层：

```
模型请求 Edit / Write
-> Permission / Sandbox / Policy 检查
-> allow / ask / deny
-> 允许后才真正写盘
```

因此正确分工是：

- **Prompt**：告诉模型在什么情况下应谨慎、应先询问。
- **Permission**：强制决定工具是否可以执行。
- **Sandbox**：限制即使工具执行后能够触及的资源范围。
- **Hooks / 审计**：补充策略、记录和拦截。

一句话：prompt 负责“劝模型守规矩”，Permission 负责“即使它不守规矩也做不到”。

**7. 如何判断能力应设计成 Tool、Skill、Subagent 还是 Side Query？**

可以用下面的判断表：

|类型|适合什么|典型例子|
|---|---|---|
|Tool|原子、可执行、需要结构化输入输出或权限控制的动作|Read、Edit、Bash、WebFetch|
|Skill|可复用的领域工作流、知识和操作指南|发布流程、特定框架开发、代码审查规范|
|Subagent|需要独立上下文、多轮推理、角色隔离或可并行的大任务|Explore、验证、复杂调研|
|Side Query|窄任务、后台辅助、结构化结果，不应污染主对话|记忆筛选、压缩、标题生成、权限解释|

一个实用判断：

```
要执行外部动作？               -> Tool
主要是教模型如何完成一类任务？ -> Skill
需要独立多轮 Agent loop？      -> Subagent
只是后台做一次分类/总结/选择？ -> Side Query
```

不要把工具做成大段操作教程，也不要把 Skill 做成唯一能调用的执行接口。Tool 是能力，Skill 是方法论。

**8. 如何评估 prompt 是否有效，而不只是“感觉回答不错”？**

评估重点应从“文本好不好看”变成“行为是否达标”。

至少建立五类指标：

```
任务成功率：是否完成用户目标
工具行为：是否选对工具、参数是否合法、是否少走弯路
安全性：是否越权、是否错误修改、是否抵抗 prompt injection
效率：token、轮数、工具调用数、延迟、缓存命中
稳定性：同类任务多次运行是否一致，改 prompt 后是否回归
```

具体做法：

- 建立真实任务集和明确验收条件，不只看模型最终文本。
- 回放历史 transcript，比对旧 prompt 与新 prompt 的工具调用轨迹。
- 为高风险流程写对抗用例：越权写文件、伪造工具结果、恶意文档指令等。
- 对 side query 使用 JSON Schema，并评估解析成功率和字段正确率。
- 记录 prompt 版本、模型版本、token、缓存命中、失败原因。
- 做 A/B 对比和人工盲评，重点查看用户是否需要纠正模型。

一个成熟的结论是：**好的 prompt 不是让模型“说得更像人”，而是让系统在正确性、安全性、成本和稳定性上可测量地变好。**
