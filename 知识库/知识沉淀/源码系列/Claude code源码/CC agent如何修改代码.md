## 一、前置介绍
大模型不具备直接修改本地代码的能力，大模型通过借助工具实现对本地代码的修改。而大模型能做的是接收一段文本代码和修改要求，然后输出一段文本代码给到工具，由工具进行本地代码的修改
## 二、三种修改代码的方案进化史
#### 2.1 第一种：全文件重写
**描述**：给大模型输入整个代码文件，然后大模型输出修改后的整个文件，工具直接覆盖原有文件全部内容即可
**优势**：工具层处理简单，仅全文替换即可，无需任何定位逻辑
**劣势**：
- **浪费 token**：假设文件有上千行，此时仅需要修改某一行代码，需要接收和输出整个文件
- **截断风险**：对于上千行的代码文件，越长的输出越容易出现中途出错或者被截断的概率，大模型原样输出代码出错概率大

#### 2.2 第二种：Diff 或叫 Patch 补丁
**描述**：由模型输出类似 diff 的格式，标明在第几行、删除了什么、增加了什么，然后由工具负责把  Patch 给打到文件上
**优势**：解决了大模型一次输入、输出大量文件代码，导致出错的问题
**劣势**：大模型会出现行号数错，尤其是文件比较长的时候或者对话中的文件已经改动过几次后，不大模型记忆中的行号和文件实际行号容易对不上。最终导致要么 Patch 应用失败，要么应用到了错误的位置上面

#### 2.3 第三种：字符串精确匹配替换
**描述**：模型不再提供指定行号，而是提供两段文本。一段是“文件里原本的代码片段”，另一段是“最终修改后的文档片段”。然后工具拿着“原文”去文件里面做精确字符串匹配，找到唯一一处匹配值，再进行替换。找不到或匹配到多处则报错，由模型继续提供更唯一的片段匹配
**优势**：将“定位代码”这件事，从“数行数”变成“认内容”。模型擅长复制看到过的一段文字，而不擅长数行号
**劣势**：
- **行号污染**：阅读代码的工具通常为了让模型方便阅读，会在查询结果上带上行号前缀。假设模型原样将带有行号的前缀输出给工具，会导致工具精确匹配时无法找到匹配值
- **文件视图过期**：假设对话很长，模型对文件内容的印象是十轮对话之前的，且在这期间文件内容被编辑过了。此时如果还要继续编辑同一个文件，最好的做法是：必须重新查看一次目前文件的内容以及每次编辑成功后都给文件视图给“过期”掉
- **新建和编辑分开**：这算是经验之谈，将“新建文件”和“编辑文件”实现为两个工具，避免出现模型以为是在“编辑”，但实际上是整个**覆盖**掉一个已经存在、有内容的文件
## 三、假设自己设计修改代码的工具
#### 3.1 代码编辑工具
把上面这些经验揉在一起呢，一个合理的编辑工具签名大概是这样的：
```javascript
str_replace(path, old_str, new_str, description)
```
- **path** : 是用绝对路径，来减少"到底是哪个文件"的歧义
- **old_str**：**必须在文件中唯一匹配**。匹配失败的时候，不管是不存在还是存在多处，工具都要明确返回失败原因，引导模型带更多上下文重试
- **new_str**：是替换后的内容，允许为空，等价于删除这段内容
- **description**：是说明这次编辑的目的，是去"逼"模型在动手之前，**先把"我为什么要这么改"给讲清楚**。某种意义上呢，这就是把思维链嵌进了工具调用本身，间接提升了编辑的准确率
#### 3.2 代码新建工具
专门"新建文件"的工具，遇到路径已存在就直接报错，绝不静默覆盖
#### 3.3 代码查询工具
支持看整个目录结构，也支持看某个文件的内容，带行号方便引用。还支持只看某个范围，避免大文件全量塞进上下文

## 四、全文升华总结
从全文件重写，到行号 diff，再到字符串精确匹配。这条演进路径呢，其实是在反复回答同一个问题。那就是怎么让"擅长语义、不擅长精确计数"的模型，和"只认精确指令、不容任何模糊"的文件系统，好好协作。

答案呢，**不是让模型变得更精确**。这个很难，也不该是解法。而是设计一层工具，让模型只需要发挥它最擅长的能力，也就是理解和复现内容。剩下需要绝对精确的部分呢，交给代码逻辑去把关、去拒绝、去要求重试。

这大概也是所有 Agent 工具设计背后共通的思路吧。**那就是不是教会模型做它不擅长的事，而是把它不擅长的事，从它的任务清单里面给挪走**
